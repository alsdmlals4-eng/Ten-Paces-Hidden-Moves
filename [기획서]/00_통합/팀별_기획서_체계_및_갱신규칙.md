# 팀별 기획서 체계 및 지속 갱신 규칙

## 1. 목적

십보강호의 모든 기획·구현·자산·검수 작업을 다음 다섯 책임 문서로 구분한다.

1. 게임 기획서
2. 프로그래밍 기획서 — 로드맵·MVP 포함
3. 아트 기획서
4. 사운드 기획서
5. QA·PM 기획서

상세 규칙의 최종 원본은 `docs/`의 책임 문서와 실제 구현 파일이다. `[기획서]/`의 다섯 책임 문서는 현재 상태, 결정, 구현 경로, 산출물, 완료 기준과 연결 문서를 사람이 빠르게 확인하는 팀별 진입점이다.

## 2. 팀별 역할

### 게임 기획팀 — Game Design

게임의 규칙, 시나리오, UI/UX, 전투, 성장, 콘텐츠, 밸런스와 플레이어 경험을 설계한다. 모든 팀이 같은 규칙을 기준으로 작업하도록 용어, 수치, 상태, 화면 흐름과 예외를 문서화한다.

책임 문서: [`../01_게임기획/게임기획서.md`](../01_게임기획/게임기획서.md)

### 프로그래밍 팀 — Programming

게임 기획을 실제 실행 가능한 시스템으로 구현한다. 십보강호의 현재 엔진은 Godot 4.7.1이며, 전투 해상, AI, 데이터, UI 연결, 저장, 툴과 자동 테스트를 담당한다. 로드맵, MVP 범위, 구현 순서, 기술 위험과 완료 조건을 함께 관리한다.

책임 문서: [`../06_프로그래밍/프로그래밍_기획서_로드맵_MVP.md`](../06_프로그래밍/프로그래밍_기획서_로드맵_MVP.md)

### 아트 팀 — Art

2D 그래픽, 캐릭터·문파·배경 디자인, 무공서, 무공 카드, 문양·아이콘, UI 장식, 애니메이션과 VFX의 시각 기준을 관리한다. 생성 이미지의 글자는 기준 데이터가 아니며, 정확한 명칭·분류·수치는 편집 가능한 UI 텍스트와 데이터로 합성한다.

책임 문서: [`../04_아트디자인/아트_기획서.md`](../04_아트디자인/아트_기획서.md)

### 사운드 팀 — Sound

BGM, 환경음, UI음, 전투 SFX, 무공·절초·진의의 청각 피드백과 향후 음성 범위를 설계한다. 사운드는 전투 결과를 결정하지 않고 구조화된 게임 이벤트를 표현한다.

책임 문서: [`../07_사운드/사운드_기획서.md`](../07_사운드/사운드_기획서.md)

### QA·PM 팀 — Quality Assurance / Project Management

QA는 규칙·구현·표현의 불일치와 버그를 재현 가능한 방식으로 검증한다. PM은 범위, 우선순위, 일정, 의존성, 위험, 승인 상태, 팀 간 인수인계와 문서 동기화를 관리한다.

책임 문서: [`../08_QA_PM/QA_PM_기획서.md`](../08_QA_PM/QA_PM_기획서.md)

## 4. 모든 작업의 필수 갱신 원칙

대화, 기획, 코드, 아트, 사운드, 테스트 중 하나라도 프로젝트의 확정 상태를 바꾸면 **같은 작업과 같은 PR에서 영향을 받은 책임 문서를 갱신한다.**

- 규칙·용어·수치·화면 흐름 변경 → 게임 기획서
- MVP 범위·구현 구조·데이터·엔진·로드맵 변경 → 프로그래밍 기획서
- 캐릭터·배경·카드·아이콘·VFX·아트 파이프라인 변경 → 아트 기획서
- BGM·SFX·보이스·오디오 이벤트·믹싱 변경 → 사운드 기획서
- 완료 기준·테스트·일정·위험·우선순위·인수인계 변경 → QA·PM 기획서

복수 분야 작업은 관련된 문서를 모두 갱신한다. 예를 들어 절초 카드 제작은 게임 기획서의 해금 규칙, 아트 기획서의 표찰·카드 규격, 프로그래밍 기획서의 데이터 연결, 사운드 기획서의 발동 피드백, QA·PM 기획서의 검수 항목에 영향을 줄 수 있다.

## 5. 작업 완료 게이트

다음 항목이 끝나지 않으면 작업 완료로 보고하지 않는다.

1. 최신 사용자 확정 내용을 책임 문서에 반영한다.
2. 실제 구현·자산 상태와 문서의 완료 표기가 일치하는지 확인한다.
3. `[기획서]/manifest.json`, `[기획서]/README.md`, 필요한 경우 `docs/ACTIVE_CONTEXT.md`와 `docs/DOCUMENTATION_MAP.md`를 갱신한다.
4. 폐기된 용어·수치·구조를 검색해 활성 문서의 충돌을 제거한다.
5. 관련 테스트와 체크리스트를 갱신한다.
6. `python scripts/check_planning_sync.py`와 `python tests/static_check.py`를 실행한다.
7. Godot 실행 환경이 있으면 headless 테스트와 수동 플레이테스트를 수행한다.
8. 실행하지 못한 검증과 남은 위험을 명시한다.

## 6. 상태 표기

각 책임 문서의 항목은 다음 상태 중 하나로 구분한다.

- **확정**: 사용자 승인과 기준 문서 반영이 끝난 상태
- **진행 중**: 방향은 승인됐으나 세부 설계·구현·자산이 미완성인 상태
- **검증 필요**: 구현 또는 자산은 있으나 실행·화면·밸런스 검증이 부족한 상태
- **보류**: 활성 구현 대상이 아니며 사용자 재승인 전까지 작업하지 않는 상태

## 7. 단일 기준 원칙

- 활성 문서를 `v2`, `final`, `latest`로 복제하지 않는다.
- 이전 상태는 Git 이력으로 보존하고 현재 기준 파일을 직접 갱신한다.
- 이미지 속 글자나 임시 수치는 기준 데이터가 아니다.
- 무공 이름·분류·숙련 해금·카드 표시 같은 정확 정보는 게임 기획 데이터와 편집 가능한 UI 레이어가 소유한다.
- 팀별 책임 문서는 현재 상태를 요약하고 상세 수치는 해당 `docs/` 기준 문서를 참조한다.
