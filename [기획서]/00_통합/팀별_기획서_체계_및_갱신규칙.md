# 팀별 기획서 체계 및 지속 갱신 규칙

## 1. 핵심 원칙

사용자가 처음 보는 활성 기획서는 정확히 다섯 개만 둔다.

1. 게임 기획서
2. 프로그래밍 기획서 — 로드맵·MVP 포함
3. 아트 기획서
4. 사운드 기획서
5. QA·PM 기획서

이 다섯 문서만 읽어도 게임 방향, 현재 상태, 품질 기준, 최신 이미지 방향과 다음 작업을 이해할 수 있어야 한다.

기존 `docs/`, 기존 `[기획서]/` 세부 문서와 `plans/`는 사용자가 찾아다녀야 하는 별도 활성 기획서가 아니다. 다섯 기획서 뒤에서 특정 수치·데이터·구현 계약·테스트를 증명하는 **상세 명세·부록**으로 사용한다.

## 2. 기획서 품질 기준

### 2.1 사용자 가독성

- 문서 첫 화면에서 프로젝트 한 줄 설명, 현재 상태, 핵심 결정과 다음 작업이 보인다.
- 긴 설명은 제목, 표, 짧은 문단, 상태 구분으로 나눈다.
- 내부 용어를 처음 사용할 때 뜻을 설명한다.
- 확정·진행 중·검증 필요·보류를 명확히 구분한다.
- 사용자가 다른 문서를 열지 않아도 핵심 방향을 이해할 수 있게 작성한다.

### 2.2 새 작업자의 재현 가능성

새로운 채팅이나 작업자는 과거 대화를 볼 수 없다고 가정한다. 각 활성 기획서는 다음 내용을 충분히 상세하게 포함해야 한다.

- 작품의 핵심 경험과 목표 품질.
- 현재 확정된 규칙·용어·구조.
- 팀별 제작 방식과 금지 사항.
- 현재 구현·자산·검증 상태.
- 파일 경로와 상세 부록 링크.
- 완료 기준과 검수 항목.
- 다음 작업의 우선순위와 미확정 범위.

문서를 읽은 작업자가 기존과 같은 방향과 품질의 결과물을 만들 수 없다면 문서가 부족한 것으로 본다.

### 2.3 최신 이미지와 방향성

- 최신 이미지의 책임 원본은 **아트 기획서와 이미지 인덱스 하나**로 고정한다.
- 게임·UI·프로그래밍 문서에서는 이미지를 복제하지 않고 원본을 링크한다.
- 이미지 생성·수정·승인·폐기 작업은 같은 작업에서 아트 기획서와 이미지 인덱스를 갱신한다.
- 이미지 속 한글·수치·분류는 기준 데이터가 아니며 편집 가능한 UI 레이어가 소유한다.
- 승인된 이미지 원본이 저장소에 없으면 `원본 반입 필요`로 명시한다.
- 오래된 시안은 `레이아웃 참고` 또는 `구식 참고`로 낮추고 신규 제작 기준으로 사용하지 않는다.

## 3. 팀별 역할과 활성 문서

### 게임 기획팀 — Game Design

게임의 규칙, 시나리오, UI/UX, 전투, 성장, 콘텐츠, 밸런스와 플레이어 경험을 설계한다. 모든 팀이 같은 규칙을 기준으로 작업하도록 용어, 수치, 상태, 화면 흐름과 예외를 문서화한다.

활성 문서: [`../01_게임기획/게임기획서.md`](../01_게임기획/게임기획서.md)

### 프로그래밍 팀 — Programming

게임 기획을 실제 실행 가능한 시스템으로 구현한다. 현재 엔진은 Godot 4.7.1이며, 전투 해상, AI, 데이터, UI 연결, 저장, 툴과 자동 테스트를 담당한다. 로드맵, MVP 범위, 구현 순서, 기술 위험과 완료 조건을 함께 관리한다.

활성 문서: [`../06_프로그래밍/프로그래밍_기획서_로드맵_MVP.md`](../06_프로그래밍/프로그래밍_기획서_로드맵_MVP.md)

### 아트 팀 — Art

2D 그래픽, 캐릭터·문파·배경 디자인, 무공서, 무공 카드, 문양·아이콘, UI 장식, 애니메이션과 VFX의 시각 기준을 관리한다.

활성 문서: [`../04_아트디자인/아트_기획서.md`](../04_아트디자인/아트_기획서.md)

최신 이미지 인덱스: [`../04_아트디자인/이미지_인덱스.md`](../04_아트디자인/이미지_인덱스.md)

### 사운드 팀 — Sound

BGM, 환경음, UI음, 전투 SFX, 무공·절초·진의의 청각 피드백과 향후 음성 범위를 설계한다. 사운드는 전투 결과를 결정하지 않고 구조화된 게임 이벤트를 표현한다.

활성 문서: [`../07_사운드/사운드_기획서.md`](../07_사운드/사운드_기획서.md)

### QA·PM 팀 — Quality Assurance / Project Management

QA는 규칙·구현·표현의 불일치와 버그를 재현 가능한 방식으로 검증한다. PM은 범위, 우선순위, 일정, 의존성, 위험, 승인 상태, 팀 간 인수인계와 문서 동기화를 관리한다.

활성 문서: [`../08_QA_PM/QA_PM_기획서.md`](../08_QA_PM/QA_PM_기획서.md)

## 4. 기존 문서의 역할

| 분야 | 상세 명세·부록 |
|---|---|
| 게임 기획 | `docs/01_GAME_DESIGN.md`, `02_COMBAT_RULES.md`, `03_CONTENT_CATALOG.md`, `06_STARTING_FACTION_MASTERY_DATA.md`, `07_COMBAT_UI_SPEC.md`, 기존 게임·전투·UI/UX 문서 |
| 프로그래밍 | `docs/04_ROADMAP.md`, `05_COMBAT_POC_SPEC.md`, `09_COMBAT_SYSTEM_ARCHITECTURE.md`, `plans/`, 실제 코드·씬·테스트 |
| 아트 | `docs/07_COMBAT_UI_SPEC.md`, `10_COMBAT_PRESENTATION_PLAN.md`, 기존 아트 가이드와 이미지 자산 |
| 사운드 | `docs/10_COMBAT_PRESENTATION_PLAN.md`의 오디오 요구사항과 실제 오디오 자산·이벤트 계약 |
| QA·PM | `docs/08_TEST_CHECKLIST.md`, `ACTIVE_CONTEXT.md`, `DOCUMENTATION_MAP.md`, 기존 플레이테스트 계획 |

장기적으로 중복 내용을 다섯 활성 기획서로 흡수한다. 고유한 수치·데이터·계약·테스트가 없는 문서만 참조 검사를 거쳐 아카이브하거나 제거한다.

## 5. 모든 작업의 필수 갱신 규칙

대화, 기획, 코드, 아트, 사운드, 테스트 중 하나라도 프로젝트의 확정 상태를 바꾸면 **같은 작업과 같은 PR에서 영향을 받은 활성 기획서를 갱신한다.**

- 규칙·용어·수치·화면 흐름 변경 → 게임 기획서
- MVP 범위·구현 구조·데이터·엔진·로드맵 변경 → 프로그래밍 기획서
- 캐릭터·배경·카드·아이콘·VFX·이미지 승인 변경 → 아트 기획서 + 이미지 인덱스
- BGM·SFX·보이스·오디오 이벤트·믹싱 변경 → 사운드 기획서
- 완료 기준·테스트·일정·위험·우선순위·인수인계 변경 → QA·PM 기획서

복수 분야 작업은 관련 문서를 모두 갱신한다.

## 6. 작업 완료 게이트

다음 항목이 끝나지 않으면 작업 완료로 보고하지 않는다.

1. 최신 사용자 확정 내용을 관련 활성 기획서에 반영한다.
2. 실제 구현·자산 상태와 문서의 완료 표기가 일치하는지 확인한다.
3. 이미지 작업이면 아트 기획서와 이미지 인덱스를 갱신한다.
4. `[기획서]/manifest.json`, `[기획서]/README.md`, 필요한 경우 `docs/ACTIVE_CONTEXT.md`와 `docs/DOCUMENTATION_MAP.md`를 갱신한다.
5. 폐기된 용어·수치·구조를 검색해 활성 문서의 충돌을 제거한다.
6. 관련 상세 명세와 테스트 체크리스트를 갱신한다.
7. `python scripts/check_planning_sync.py`와 `python tests/static_check.py`를 실행한다.
8. Godot 실행 환경이 있으면 headless 테스트와 수동 플레이테스트를 수행한다.
9. 실행하지 못한 검증과 남은 위험을 명시한다.

## 7. 단일 기준 원칙

- 활성 기획서를 `v2`, `final`, `latest`로 복제하지 않는다.
- 이전 상태는 Git 이력으로 보존하고 현재 파일을 직접 갱신한다.
- Documentation Map 첫 화면에는 다섯 활성 기획서만 보여준다.
- 각 활성 기획서 안에 분야별 목차와 상세 부록 링크를 둔다.
- 한 규칙의 소유권은 한 문서에 두고 다른 문서는 링크한다.
- 구현·자산·테스트를 확인하지 않은 상태는 완료로 쓰지 않는다.
