# 십보강호 QA·PM 기획서

## 0. 한눈에 보기

QA는 게임 규칙·구현·화면·아트·사운드가 기획대로 작동하는지 검증하고, PM은 범위·우선순위·일정·위험·승인·팀 간 인수인계를 관리한다.

현재 프로젝트는 **T0 전투 프로토타입 실행 준비와 문서 체계 정비 단계**다. 정적 검사와 문서 동기화 검사는 존재하지만 Godot 런타임과 Windows 수동 플레이테스트는 아직 완료로 확정하지 않는다.

## 1. QA·PM 팀의 책임

### QA

- 재현 가능한 버그 보고서를 작성한다.
- 규칙과 구현의 불일치를 찾는다.
- 기능, UI/UX, 밸런스, 아트, 사운드, 접근성, 저장 안정성을 검증한다.
- 자동 테스트와 수동 플레이테스트의 범위를 구분한다.
- 실행하지 못한 검증을 완료로 표시하지 않는다.

### PM

- 다섯 활성 기획서의 최신 상태를 유지한다.
- 작업 범위, 우선순위, 의존성, 일정, 위험과 승인 상태를 관리한다.
- 변경이 여러 팀에 미치는 영향을 확인한다.
- 사용자 최신 결정을 기준 문서와 구현에 같은 작업으로 반영한다.
- 새 작업자가 저장소만 읽고 기존과 같은 품질로 작업하도록 인수인계를 유지한다.

## 2. 활성 기획서 게이트

사용자가 처음 보는 활성 기획서는 정확히 다섯 개다.

1. 게임 기획서
2. 프로그래밍 기획서 — 로드맵·MVP
3. 아트 기획서
4. 사운드 기획서
5. QA·PM 기획서

기존 `docs/`와 세부 `[기획서]/` 문서는 상세 명세·부록으로 사용한다. 첫 화면에서 별도 활성 기획서처럼 나열하지 않는다.

### 문서 품질 기준

각 활성 기획서는 다음을 만족해야 한다.

- 처음 읽는 사용자가 핵심 방향과 현재 상태를 빠르게 이해한다.
- 새로운 채팅·작업자가 대화 기록 없이 같은 품질로 작업할 정도로 상세하다.
- 확정, 진행 중, 검증 필요, 보류가 구분된다.
- 현재 산출물, 금지 사항, 다음 작업과 부록 링크가 있다.
- 아트 문서는 최신 이미지 인덱스를 연결하고 오래된 이미지 복제를 막는다.
- 용어·수치·이미지 방향이 바뀌면 같은 작업에서 최신화된다.

## 3. 테스트 계층

### 3.1 문서·정적 검사

```bash
python scripts/check_planning_sync.py
python tests/static_check.py
```

검증 대상:

- 다섯 활성 기획서 존재.
- manifest의 활성 문서가 정확히 다섯 개.
- README와 문서 지도에서 다섯 문서만 첫 진입점으로 노출.
- frontmatter와 링크 유효성.
- 이미지 인덱스와 아트 기획서 연결.
- 실제 구현 경로 존재.
- 폐기된 용어·수치 검색.

### 3.2 자동 전투 테스트

```bash
godot --headless --path . --script res://tests/run_all.gd
```

검증 대상:

- AI 선잠금.
- 두 수 동시 공개·순차 해상.
- 위치·거리.
- 합과 차이 피해.
- 자원 부족과 사거리 실패.
- 1수 사망 시 2수 처리.
- 절초 기세 상태.
- 동일 시드 재현성.

### 3.3 수동 플레이테스트

- Windows 실제 화면과 입력.
- 1280×800 이상에서 UI 겹침.
- 카드와 팝업 가독성.
- 합·일반 공격·막기·회피 인지.
- 절초 기세 증가 이유와 사용 가능 상태.
- 이미지·아이콘·텍스트 매핑 오류.
- 오디오 정보성과 반복 피로.

## 4. T0 플레이테스트 계획

### 환경

- Godot 4.7.1 stable.
- Windows 우선.
- 권장 창 1280×800 이상.

### 진행

1. 설명 없이 첫 전투를 관찰한다.
2. 툴팁과 로그를 허용한 두 번째 전투를 진행한다.
3. 절초 사용을 목표로 세 번째 전투를 진행한다.
4. 플레이테스트 기록 템플릿에 관찰과 발언을 기록한다.

### 핵심 질문

- AI가 플레이어 선택 전에 잠겼다고 믿는가?
- 두 수를 미리 계획한다는 규칙을 이해하는가?
- 합과 단독 공격을 구분하는가?
- 기세가 오른 이유와 소모 이유를 설명하는가?
- 절초가 없어도 선택이 재미있는가?
- 카드에서 문파·기술 종류·이름을 빠르게 식별하는가?

### 즉시 중단 기준

- 크래시, 진행 불가, 입력 잠김.
- AI가 후행 선택하는 것으로 보임.
- 합과 일반 공격을 구분할 수 없음.
- 행동 실패 원인을 찾을 수 없음.
- 카드 분류·문파 아이콘이 실제 데이터와 다름.
- 텍스트가 잘리거나 이미지가 UI를 가림.

## 5. 분야별 QA 체크

### 5.1 게임 기획

- 최신 규칙과 용어가 다섯 문서와 부록에서 충돌하지 않는다.
- 3·5·7·9·10성 구조가 모든 활성 문서에 일치한다.
- 태극검은 검법, 태극권은 권법으로 정확히 분류된다.
- 10성 핵심무공은 절초, 보조무공은 진의다.
- 카드 앞면과 상세 팝업의 정보 책임이 분리된다.

### 5.2 프로그래밍

- 구현 상태가 실제 파일과 테스트 결과에 맞게 기록된다.
- UI·아트·사운드가 전투 결과를 직접 계산하지 않는다.
- 데이터 ID와 자산 경로가 안정적이다.
- 누락 자산과 잘못된 데이터에 폴백·로그가 있다.
- 저장·불러오기에서 생성 결과가 재굴림되지 않는다.

### 5.3 아트

- 남녀 캐릭터가 독립 패널과 다른 포즈를 가진다.
- 문파별 색·실루엣·배경·기술 궤적이 구분된다.
- 무공 종류는 텍스트가 아니라 정확한 아이콘 이미지다.
- 생성 이미지의 오탈자가 최종 UI에 남지 않는다.
- 묘리 문구가 실제 축소 크기에서 읽힌다.
- 절초·진의가 일반 카드와 구분된다.
- 이미지 인덱스의 승인 상태와 실제 파일이 일치한다.

### 5.4 사운드

- 합·일반 공격·막기·회피가 소리로 구분된다.
- 절초 가능·예약·발동이 서로 다른 소리다.
- 소리가 숨겨진 AI 선택을 유출하지 않는다.
- BGM이 UI·판정음을 가리지 않는다.
- 중요한 상태는 시각 피드백도 함께 있다.

### 5.5 접근성

- 색상만으로 상태를 구분하지 않는다.
- 모션 감소·점멸 감소 폴백을 제공한다.
- 작은 글자와 낮은 대비를 피한다.
- 오디오 카테고리별 볼륨 조절을 고려한다.
- 키보드·마우스 입력 상태가 명확하다.

## 6. 버그 등급과 처리

| 등급 | 기준 | 처리 |
|---|---|---|
| Blocker | 실행 불가, 진행 불가, 저장 손상 | 즉시 중단·최우선 수정 |
| Critical | 전투 결과 오류, AI 공정성 위반, 주요 데이터 손상 | 배포 차단 |
| Major | 핵심 UI 오인, 카드 분류 오류, 중요한 연출·사운드 누락 | 해당 단계 승인 전 수정 |
| Minor | 경미한 정렬, 오탈자, 비핵심 시각 문제 | 우선순위에 따라 수정 |
| Suggestion | 품질 개선 제안 | PM 검토 후 백로그 |

버그 보고서에는 환경, 빌드·커밋, 재현 단계, 기대 결과, 실제 결과, 빈도, 첨부 자료와 관련 기획서 링크를 포함한다.

## 7. PM 작업 흐름

### 작업 시작

1. 다섯 활성 기획서를 읽는다.
2. 현재 작업과 연결된 상세 부록을 연다.
3. 확정·미확정·보류를 구분한다.
4. 수정 대상, 산출물, 테스트, 중단 기준을 계획한다.

### 작업 중

- 새 결정은 즉시 관련 기획서의 갱신 대상으로 표시한다.
- 다른 팀에 영향을 주면 해당 문서와 담당 산출물을 연결한다.
- 이미지 생성·승인 시 아트 기획서와 이미지 인덱스를 갱신한다.
- 임시 수치·문구는 승인 상태로 오인되지 않게 표시한다.

### 작업 종료

1. 영향받은 활성 기획서를 갱신한다.
2. 상세 부록과 실제 파일을 동기화한다.
3. manifest, README, Documentation Map, Active Context를 확인한다.
4. 폐기된 용어와 중복 내용을 검색한다.
5. 자동 검사와 가능한 런타임 테스트를 수행한다.
6. 수행하지 못한 검증과 남은 위험을 보고한다.
7. 다음 작업과 우선순위를 기록한다.

## 8. 일정·범위 관리

### 현재 우선순위

1. 다섯 활성 기획서 체계 고정.
2. 저장소 실제 실행 상태 감사.
3. T0 전투 런타임 검증.
4. 3·5·7·9·10성 구조와 10문파 데이터를 기존 문서에 동기화.
5. 무당파 아트 파이프라인 기준 자산 제작.
6. 절초 기세 규칙 확정.
7. 5전 데모 MVP 구현.

### 범위 통제

- 승인되지 않은 보류 항목은 구현하지 않는다.
- 새 문파·경제·복잡한 덱빌딩은 현재 MVP에 추가하지 않는다.
- 아트·사운드 고급 제작은 규칙과 데이터 계약이 안정된 뒤 확장한다.
- 일정이 밀리면 기능 삭제보다 검증 가능한 세로 슬라이스를 우선한다.

## 9. 현재 위험

| 위험 | 영향 | 대응 |
|---|---|---|
| 기존 문서에 12세력·3·6·9성 구조 잔존 | 작업자 혼선 | 활성 기획서 기준 고정 후 부록 순차 동기화 |
| Godot 런타임 미검증 | 구현 상태 오판 | 실제 실행 감사와 수동 테스트 우선 |
| 최신 승인 이미지 원본 미반입 | 아트 방향 유실 | 이미지 인덱스에 반입 대기 기록, 원본 확보 후 저장소 보존 |
| 이미지 생성 한글 오류 | 분류·명칭 오표기 | 텍스트·아이콘 후합성, QA 매핑 검사 |
| 상세 효과·비용 미확정 | 데이터 구현 지연 | 카드 앞면 제작과 효과 설계를 분리 |
| 사운드 자산 미착수 | 전투 피드백 부족 | T0 최소 판정음부터 프로토타입 |

## 10. 현재 상태

### 확정

- 다섯 활성 기획서 체계.
- 모든 작업에서 영향받은 기획서 갱신.
- 이미지 인덱스 단일 책임.
- 문서·정적·자동·수동 테스트 계층.

### 검증 필요

- 새 자동 검사가 다섯 문서 구조를 실제로 강제하는지 확인.
- Godot headless 실행.
- Windows 수동 플레이테스트.
- 최신 이미지 원본 저장소 반입.

## 13. 변경 시 갱신 대상

테스트 범위, 완료 기준, 일정, 우선순위, 위험, 승인 상태, 문서 구조 또는 팀 간 인수인계가 바뀌면 이 문서를 같은 작업에서 갱신한다.
