# 십보강호 QA·PM 기획서

## 0. 한눈에 보기

QA는 규칙·구현·화면·아트·사운드가 기획대로 작동하는지 검증한다. PM은 범위·우선순위·일정·위험·승인·팀 간 인수인계를 관리한다.

현재는 **다섯 활성 기획서 체계 정비 + T0 전투 프로토타입 실행 준비 단계**다. 정적 검사는 존재하지만 Godot 런타임과 Windows 수동 플레이테스트는 아직 완료로 확정하지 않는다.

## 1. QAPM 팀의 책임

### QA

- 재현 가능한 버그 보고서를 작성한다.
- 규칙과 구현의 불일치를 찾는다.
- 기능, UI/UX, 밸런스, 아트, 사운드, 접근성, 저장 안정성을 검증한다.
- 자동 테스트와 수동 플레이테스트를 구분한다.
- 실행하지 못한 검증을 완료로 표시하지 않는다.

### PM

- 다섯 활성 기획서의 최신 상태를 유지한다.
- 범위, 우선순위, 일정, 의존성, 위험과 승인 상태를 관리한다.
- 변경이 여러 팀에 미치는 영향을 확인한다.
- 사용자의 최신 결정을 같은 작업에서 문서·구현·자산·테스트에 반영한다.
- 새 작업자가 저장소만 읽고 같은 품질로 작업하도록 인수인계를 유지한다.

## 2. 다섯 활성 기획서 게이트

사용자에게 처음 보이는 활성 기획서는 정확히 다섯 개다.

1. 게임 기획서
2. 프로그래밍 기획서 — 로드맵·MVP
3. 아트 기획서
4. 사운드 기획서
5. QA·PM 기획서

기존 `docs/`, 세부 `[기획서]/`, `plans/`는 상세 명세·부록이다.

### 문서 품질 기준

각 활성 기획서는 다음을 만족해야 한다.

- 한눈에 보기와 분야별 목차가 있다.
- 사용자가 빠르게 핵심 방향과 현재 상태를 이해한다.
- 새 작업자가 과거 대화 없이 같은 품질로 작업할 만큼 상세하다.
- 확정·진행 중·검증 필요·보류를 구분한다.
- 품질 기준, 금지 사항, 현재 상태, 다음 작업과 부록 링크가 있다.
- 아트 문서는 최신 이미지 인덱스를 연결한다.
- 변경이 생기면 같은 작업과 같은 PR에서 갱신된다.

## 3. 테스트 계층

### 3.1 문서·정적 검사

```bash
python scripts/check_planning_sync.py
python tests/static_check.py
```

검증 대상:

- manifest의 활성 문서가 정확히 다섯 개.
- README와 Documentation Map이 다섯 문서만 진입점으로 노출.
- 다섯 문서의 frontmatter·목차·현재 상태·다음 작업·부록.
- 아트 기획서와 이미지 인덱스 연결.
- 이미지 링크와 구현 경로.
- 구형 12세력·3·6·9성 표현의 잔존 여부.

### 3.2 자동 전투 테스트

```bash
godot --headless --path . --script res://tests/run_all.gd
```

검증 대상:

- AI 선잠금.
- 두 수 동시 공개·순차 해상.
- 위치·거리.
- 합과 차이 피해.
- 자원 부족·사거리 실패·행동 취소.
- 절초 기세 상태.
- 동일 시드 재현성.

### 3.3 수동 플레이테스트

- Windows 실제 화면과 입력.
- 1280×800 이상에서 UI 겹침.
- 카드·묘리·팝업 가독성.
- 합·일반 공격·막기·회피 인지.
- 절초 기세 증가 이유와 사용 가능 상태.
- 문파·무공 종류 아이콘과 데이터의 매핑.
- 오디오 정보성과 반복 피로.

## 4. T0 플레이테스트

### 진행

1. 설명 없이 첫 전투를 관찰한다.
2. 툴팁과 로그를 허용한 두 번째 전투를 진행한다.
3. 절초 사용을 목표로 세 번째 전투를 진행한다.
4. 플레이테스트 기록 템플릿에 관찰과 발언을 기록한다.

### 핵심 질문

- AI가 플레이어 선택 전에 잠겼다고 믿는가?
- 두 수를 미리 계획한다는 규칙을 이해하는가?
- 합과 단독 공격을 구분하는가?
- 기세가 오른 이유와 소모 이유를 설명하는가?
- 절초가 없어도 선택이 재미있는가?
- 카드에서 문파·분류·기술 이름을 빠르게 식별하는가?

### 즉시 중단 기준

- 크래시, 진행 불가, 입력 잠김.
- AI가 후행 선택하는 것으로 보임.
- 합과 일반 공격을 구분할 수 없음.
- 행동 실패 원인을 찾을 수 없음.
- 카드 분류·문파 아이콘이 실제 데이터와 다름.
- 텍스트가 잘리거나 이미지가 UI를 가림.

## 5. 분야별 QA

### 게임 기획

- 10문파와 공식 무공 16종이 일치한다.
- 3·5·7·9·10성 구조가 모든 활성 문서에 일치한다.
- 태극검은 검법, 태극권은 권법으로 정확히 분류된다.
- 10성 핵심무공은 절초, 보조무공은 진의다.
- 카드 앞면과 상세 팝업의 정보 책임이 분리된다.

### 프로그래밍

- 구현 상태가 실제 파일과 테스트 결과에 맞다.
- UI·아트·사운드가 전투 결과를 직접 계산하지 않는다.
- 데이터 ID와 자산 경로가 안정적이다.
- 누락 데이터·자산에 폴백과 오류 로그가 있다.
- 같은 시드와 입력에서 결과를 재현한다.

### 아트

- 남녀 캐릭터가 독립 패널과 다른 포즈를 가진다.
- 문파별 색·실루엣·배경·기술 궤적이 구분된다.
- 무공 종류는 텍스트가 아니라 정확한 아이콘 이미지다.
- 생성 이미지의 오탈자가 최종 UI에 남지 않는다.
- 묘리 문구가 실제 축소 크기에서 읽힌다.
- 절초·진의가 일반 카드와 구분된다.
- 이미지 인덱스의 승인 상태와 실제 파일이 일치한다.

### 사운드

- 합·일반 공격·막기·회피가 소리로 구분된다.
- 절초 가능·예약·발동이 서로 다른 소리다.
- 소리가 숨겨진 AI 선택을 유출하지 않는다.
- BGM이 UI·판정음을 가리지 않는다.

### 접근성

- 색상만으로 상태를 구분하지 않는다.
- 모션 감소·점멸 감소 폴백을 제공한다.
- 작은 글자와 낮은 대비를 피한다.
- 오디오 카테고리별 볼륨 조절을 고려한다.

## 6. 버그와 위험 관리

### 버그 등급

| 등급 | 기준 | 처리 |
|---|---|---|
| Blocker | 실행 불가, 진행 불가, 저장 손상 | 즉시 중단·최우선 수정 |
| Critical | 전투 결과 오류, AI 공정성 위반, 주요 데이터 손상 | 배포 차단 |
| Major | 핵심 UI 오인, 카드 분류 오류, 주요 연출·사운드 누락 | 단계 승인 전 수정 |
| Minor | 경미한 정렬, 오탈자, 비핵심 시각 문제 | 우선순위에 따라 수정 |
| Suggestion | 품질 개선 제안 | PM 검토 후 백로그 |

버그 보고서에는 환경, 커밋, 재현 단계, 기대 결과, 실제 결과, 빈도, 첨부 자료와 관련 기획서 링크를 포함한다.

### 현재 주요 위험

| 위험 | 대응 |
|---|---|
| 부록에 12세력·3·6·9성 표현 잔존 | 활성 기획서 기준 고정 후 순차 동기화 |
| Godot 런타임 미검증 | 실제 실행 감사와 수동 테스트 우선 |
| 최신 이미지 원본 미반입 | 이미지 인덱스에서 반입 대기 관리 |
| 생성 이미지 한글 오류 | 텍스트·아이콘 후합성, 매핑 QA |
| 효과·비용 미확정 | 카드 앞면 제작과 밸런스 설계를 분리 |
| 사운드 자산 미착수 | T0 최소 판정음부터 제작 |

## 7. PM 작업 흐름

### 작업 시작

1. 다섯 활성 기획서를 읽는다.
2. 현재 작업과 연결된 상세 부록을 연다.
3. 확정·미확정·보류를 구분한다.
4. 산출물, 테스트, 중단 기준과 영향 문서를 정한다.

### 작업 중

- 새 결정의 문서 갱신 대상을 즉시 표시한다.
- 복수 팀 영향이 있으면 해당 문서를 모두 연결한다.
- 이미지 작업은 아트 기획서와 이미지 인덱스를 함께 갱신한다.
- 임시 수치·문구를 확정 상태처럼 사용하지 않는다.

### 작업 종료

1. 영향받은 활성 기획서 갱신.
2. 상세 부록과 실제 파일 동기화.
3. manifest·README·Documentation Map·Active Context 확인.
4. 폐기 용어와 중복 내용 검색.
5. 자동 검사와 가능한 런타임·수동 검증 수행.
6. 미수행 검증, 남은 위험, 다음 작업 기록.

## 8. 현재 상태

### 확정

- 다섯 활성 기획서 체계.
- 모든 작업에서 영향받은 문서 갱신.
- 이미지 인덱스 단일 책임.
- 문서·정적·자동·수동 테스트 계층.

### 검증 필요

- 새 자동 검사가 다섯 문서 구조를 실제로 강제하는지 확인.
- Godot headless 실행.
- Windows 수동 플레이테스트.
- 최신 이미지 원본 저장소 반입.
