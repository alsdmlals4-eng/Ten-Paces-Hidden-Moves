## 프로젝트 Trigger

이 확장을 사용한다.

- 전투·성장·문파·UI·연출 방향이나 수치가 확정될 때.
- 새 기획서·로드맵·Plan·테스트 문서를 만들거나 크게 수정할 때.
- Codex 또는 다른 작업자에게 구현을 인계할 때.
- 작업 종료 시 Active Context와 Base 학습 환류를 정리할 때.
- 기존 문서 사이의 중복·충돌·낡은 수치를 발견했을 때.

사용하지 않는다.

- 단순 오탈자처럼 방향·범위·검증에 영향이 없는 수정.
- 사용자가 재개하지 않은 `docs/[보류]/` 항목.
- 실제 파일을 읽지 않은 상태에서 구현 완료를 추정하는 작업.

## 프로젝트 추가 규칙

### 용어와 불변 구조

- 프로젝트명: 십보강호.
- 핵심 구조: 전장 10칸·행동 타이밍 10개·대회 10전.
- 데모: 1~2전 내부전, 3~4전 예선, 5전 예선 결승.
- 전투 선택: 행동 두 개를 비공개 잠금·동시 공개·1수→2수 해상.
- 상대 표현: 내부 등급·행운·수련 스타일을 이명·풍문·전적·정탐으로 암시.
- 문서 상태: 승인 계획과 실제 구현·검증을 구분.

### 읽기 순서

```text
AGENTS.md
→ docs/BASE_RULES_VERSION.md
→ docs/ACTIVE_CONTEXT.md
→ docs/DOCUMENTATION_MAP.md
→ 현재 작업의 책임 기획서
→ docs/04_ROADMAP.md
→ 현재 Plan·Issue·Goal
→ 실제 수정 대상과 연결 테스트
```

`docs/[백업]/`과 `docs/[보류]/`는 사용자가 재개하지 않는 한 기본 읽기에서 제외한다.

## 작업 절차

### 1. 의도와 상태 확인

- 최신 사용자 요청에서 해결할 문제와 플레이어 경험을 추출한다.
- 이미 승인된 결정, 미확정, 보류와 금지 범위를 분리한다.
- 실제 구현 상태가 필요한 경우 저장소 파일과 테스트를 먼저 확인한다.

### 2. 문서 영향 범위 결정

- `DOCUMENTATION_MAP.md`에서 질문의 책임 원본을 찾는다.
- 범위·완료 기준이 바뀌면 로드맵, 테스트, Plan 영향을 함께 확인한다.
- 새 문서는 기존 원본으로 표현할 수 없는 독립 책임이 있을 때만 만든다.

### 3. 승인 기획 작성

다음 순서를 사용한다.

```text
목적
→ 목표 플레이어 경험
→ 고정 규칙
→ 흐름
→ 데이터·상태 소유
→ 예외·폴백
→ 미확정
→ 로드맵·남은 작업
→ 관찰 가능한 완료 기준
```

연출·UI 시안의 임시 숫자로 전투·성장 기준을 확정하지 않는다.

### 4. 구현 인수

- Codex는 Plan Mode에서 실제 파일 상태와 문서 차이를 먼저 보고한다.
- 사용자 승인 전에는 코드·씬·Resource·저장 포맷을 수정하지 않는다.
- 구현 Plan에는 정확한 파일, 데이터 계약, 테스트, 위험과 중단 기준을 기록한다.
- 구현자가 기획을 임의로 확장하거나 `[보류]` 기능을 추가하지 않게 한다.

### 5. 작업 종료 최신화

- 확정된 결정과 수치를 책임 문서에 반영한다.
- `ACTIVE_CONTEXT.md`의 현재 상태·미확정·실행 순서를 갱신한다.
- 로드맵의 완료·진행·다음 단계와 종료 기준을 갱신한다.
- README, 문서 지도, 테스트, Plan의 링크와 용어를 확인한다.
- 폐기된 수치·표현·구조를 검색한다.
- 실제 구현·테스트를 확인하지 않은 항목은 완료로 표시하지 않는다.

### 6. Base 학습 환류

- 프로젝트 고유 이름·수치·세계관·Godot 경로를 분리한다.
- 반복 가능한 문제 해결 원리, 실패 방지와 검증 방법을 추출한다.
- 기존 Base method·skill·template·case와 중복을 확인한다.
- 한 프로젝트에서 채택한 설계는 먼저 case·가설로 기록한다.
- 실제 반복 검증이 없는 새 실행 절차를 검증된 skill로 승격하지 않는다.
- Base 갱신 결과와 미승격 이유를 `11_BASE_ADOPTION_AND_LEARNING_LOG.md`에 기록한다.

## 산출물

작업 유형에 따라 다음 중 필요한 것만 갱신한다.

- 분야별 책임 기획서.
- `ACTIVE_CONTEXT.md`.
- `DOCUMENTATION_MAP.md`.
- `04_ROADMAP.md`.
- `08_TEST_CHECKLIST.md`.
- 현재 Codex Plan.
- `BASE_RULES_VERSION.md`.
- `11_BASE_ADOPTION_AND_LEARNING_LOG.md`.
- Base method·skill·template·case와 Changelog.

## 프로젝트 전용 완료 기준

- [ ] 기획서만 읽어도 핵심 경험, 현재 방향, 포함·제외 범위를 이해한다.
- [ ] 더 자세한 규칙과 실제 파일의 책임 원본 경로가 명시돼 있다.
- [ ] 로드맵에 현재 단계, 선행 조건, 다음 작업과 종료 기준이 있다.
- [ ] 승인·미확정·구현·검증·보류 상태가 구분된다.
- [ ] UI·연출·도메인·저장의 책임이 섞이지 않는다.
- [ ] 작업자가 과거 대화 없이 다음 행동을 선택한다.
- [ ] 프로젝트 전용 값과 Base 공용 학습 데이터가 구분된다.
- [ ] 구현하지 않은 기능을 완료로 보고하지 않는다.

## 프로젝트 전용 검증

### 콜드 스타트 질문

새 작업자에게 다음을 확인한다.

1. 십보강호의 핵심 재미와 데모 범위는 무엇인가?
2. 지금 구현을 시작해도 되는가?
3. 전투 규칙, UI, 연출과 로드맵의 책임 문서는 어디인가?
4. 다음 작업과 구현 전 남은 결정은 무엇인가?
5. 개발용 상대 태그를 플레이어에게 어떻게 변환하는가?
6. `[보류]`에서 구현하면 안 되는 내용은 무엇인가?
7. 이번 작업에서 Base로 환류한 내용과 프로젝트에 남긴 값은 무엇인가?

10분 안에 정확히 답하지 못하면 Active Context, 문서 지도, 로드맵 또는 이 확장을 보강한다.

### 정적 검색

- 구 6슬롯·1~5성·9전·구 성과 보상 용어.
- `[강적] [A++] [올인형]` 같은 개발용 태그의 플레이어 UI 노출.
- UI·연출 코드의 피해·보상·수련·저장 직접 변경.
- `v2`, `final`, `latest`, 날짜별 활성 복제 문서.
- 활성 문서의 `[보류]` 기능 구현 지시.

## 승격 검토

Base로 일반화 가능한 교훈:

- 내부 데이터와 세계관 표현의 의미 키 분리.
- 공용 Base 채택과 프로젝트 검증·환류를 한 작업 종료 순환으로 관리.
- 연출 책임 원본에 향후 로드맵·남은 작업·폴백·상태 소유를 포함.

프로젝트에 남길 고유 내용:

- 10-10-10 구조와 5전 데모.
- 문파·무공·제약·밸런스 수치.
- 이명·풍문 실제 문장과 데이터 키.
- Godot 파일·씬·Resource·테스트 경로.
