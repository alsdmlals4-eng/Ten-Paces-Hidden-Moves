# 십보강호 프로그래밍 기획서 — 로드맵과 MVP

## 0. 한눈에 보기

현재 엔진은 **Godot 4.7.1 stable**이다. 10칸 전장, 두 수 선잠금, 합, 자원, 절초 기세를 먼저 검증한 뒤 5전 데모와 10전 전체 대회로 확장한다.

현재 브랜치의 핵심 경로:

- `project.godot`
- `scenes/combat_playtest.tscn`
- `src/combat/combat_simulator.gd`
- `src/ui/combat_playtest.gd`
- `tests/run_all.gd`

파일과 정적 검사는 존재하지만 Godot 런타임과 Windows 수동 플레이테스트는 **검증 필요**다.

## 1. 프로그래밍 팀의 책임

- 게임 기획을 실행 가능한 시스템·데이터·UI로 구현한다.
- AI가 플레이어 선택을 본 뒤 대응하지 못하도록 선잠금 시점을 보장한다.
- 같은 시드와 입력에서 같은 결과를 재현한다.
- 전투 결과와 프레젠테이션을 분리한다.
- 이미지 속 임시 한글·수치·분류를 기준 데이터로 사용하지 않는다.
- 프로토타입 존재, 구현 완료, 런타임 검증 완료를 구분한다.

## 2. 시스템 아키텍처

### 2.1 전투 도메인 소유권

전투 도메인이 소유한다.

- 위치와 거리
- 행동력·기력·내공
- 행동 잠금과 공개 순서
- 공격 유효성
- 합 수치와 피해
- 막기·회피·밀치기·상태
- 절초 기세 획득·상한·소모
- 승패와 라운드 종료
- 성장과 저장 결과

UI·애니메이션·VFX·사운드는 구조화 이벤트를 구독해 표현만 한다.

예시 이벤트:

`actions_locked / actions_revealed / movement_resolved / attack_resolved / clash_started / clash_resolved / resource_changed / ultimate_gauge_changed / action_cancelled / round_ended`

### 2.2 무공 데이터 계약

무공 데이터가 소유할 값:

- 문파·무공서·기술의 안정적인 ID
- 무공 분류
- 성급 해금 조건
- 사거리·대상·비용
- 기본 효과·특수효과·키워드
- 5성·9성 강화값
- 카드 아트·문파 문양·분류 아이콘 경로
- 묘리 문구와 절초·진의 표찰 타입

정확한 이름·사거리·비용·검법/권법 구분은 UI 텍스트와 아이콘 자산으로 합성한다.

### 2.3 오류와 폴백

- 누락 데이터는 크래시보다 명확한 오류 로그와 공용 폴백을 사용한다.
- 자산이 없어도 전투 규칙 테스트가 가능해야 한다.
- 문자열 이름 비교보다 ID 참조를 우선한다.
- UI·VFX·사운드가 상태를 직접 변경하지 못하게 한다.

## 3. MVP 정의

### 3.1 T0 — 핵심 전투 PoC

목적: 핵심 판단과 AI 공정성을 검증한다.

포함:

- 10칸 직선 전장
- AI 두 수 선잠금
- 플레이어 두 수 잠금
- 양측 동시 공개
- 1수→2수 순차 해상
- 접근·후퇴·공격·막기·회피·명상
- 합과 차이 피해
- 자원 부족·사거리 실패·행동 취소
- 절초 기세 프로토타입
- 구조화 로그와 전투 재시작

제외:

- 5전 대회 흐름
- 수련·성과·행운
- 상대 이명·풍문·정탐
- 10문파 전체 효과 데이터
- 저장·불러오기
- 고급 아트·애니메이션·사운드

완료 기준:

- AI가 플레이어 입력 전에 잠긴다.
- 첫 수 뒤 둘째 수를 바꿀 수 없다.
- 이동 뒤 거리로 공격 유효성을 판정한다.
- 합과 일반 공격 결과가 분리된다.
- 같은 시드·입력으로 결과를 재현한다.
- 행동 성공·실패 이유를 로그에서 설명할 수 있다.

### 3.2 T1 — 5전 데모 MVP

- 1~2전 내부전, 3~4전 예선, 5전 예선 결승.
- 문파 선택과 시작 무공서.
- 전투 후 성과·행운·수련.
- 3·5·7·9·10성 성장.
- 3성·7성 신규 기술, 10성 절초·진의.
- 상대 성장과 재현 가능한 생성.
- 이명·풍문·전적·정탐.
- 기본 저장·불러오기.

대표 문파 소수로 파이프라인을 검증한 뒤 10문파로 확장한다.

### 3.3 T2 — 전체판 기반

- 10전 대회.
- 10문파, 문파 전용 무공서 28개.
- 문파 전용 카드 84장.
- 공용 무공.
- 상대 풀과 수련 스타일.
- 고급 연출·사운드·접근성.
- 밸런스·콘텐츠 검증 툴.

## 4. 구현 로드맵

### 단계 0 — 기준선과 감사

- 다섯 활성 기획서와 연결 부록을 읽는다.
- 실제 파일과 문서의 차이를 감사한다.
- 미확정 수치를 승인 없이 영구값으로 굳히지 않는다.

### 단계 1 — 전투 도메인

위치·거리, 행동 잠금, 해상 순서, 합, 자원, 실패·취소, 결정론적 테스트.

### 단계 2 — 전투 UI

10칸 전장, 카드 선택, 1수·2수 슬롯, AI 잠금, 동시 공개, 합·피해·자원 로그, 절초 기세 HUD.

### 단계 3 — 무공 데이터

16종 분류, 10문파·28개 무공서, 3·7·10성 카드, 5·9성 강화, 문파 문양·분류 아이콘 연결.

### 단계 4 — 5전 데모

대회 진행, 상대 생성, 성과·행운, 수련, 저장·불러오기, 예선 결승.

### 단계 5 — 제작 파이프라인

카드 데이터·아트·사운드 누락 검사, 텍스트·아이콘 합성, 밸런스 시뮬레이션.

## 6. 구현 품질 기준

- 전투 결과와 표현을 분리한다.
- 이벤트 이름과 데이터 계약을 문서화한다.
- UI는 1280×800 이상에서 겹침 없이 읽혀야 한다.
- 데이터·자산 누락에 폴백과 오류 로그가 있다.
- 모션 감소·점멸 감소·색상 외 상태 표시를 고려한다.
- 테스트 없이 구현 완료로 표기하지 않는다.
- 최신 게임·아트·사운드·QA 기획서의 계약을 따른다.

## 7. 테스트 명령

```bash
python scripts/check_planning_sync.py
python tests/static_check.py
godot --headless --path . --script res://tests/run_all.gd
```

Godot 바이너리가 없으면 앞의 두 검사를 수행하고 런타임 미검증을 기록한다.
