# 십보강호 AI 협업 흐름

## 역할

### GPT·기획 작업

- 사용자 의도·플레이어 경험·범위·제약·완료 기준을 정리한다.
- 책임 원본과 상태를 갱신한다.
- L2 이상 구조·워크플로 변경은 저장소 사실과 사용자 확인을 기록한다.
- 구현되지 않은 기능을 완료로 쓰지 않는다.

### Codex·구현 작업

- 실제 로컬 저장소·Git 상태·파일·테스트를 먼저 감사한다.
- 승인 Plan의 정확한 파일·데이터 계약·테스트만 수정한다.
- 사용자 변경과 저장 호환성을 보호한다.
- 구현·검증 결과를 문서와 Handoff에 반영한다.

### 외부 조사·대량 초안

- 격리된 작업 공간에서 수행한다.
- 결과는 프로젝트 실제 파일·책임 원본·검증과 대조한 뒤 반영한다.
- 환각 경로·미확인 수치·중복 본책을 그대로 병합하지 않는다.

### GitHub

- Issue·직접 요청·Plan을 작업 계약으로 사용한다.
- 브랜치와 Draft PR에서 변경을 격리한다.
- PR 본문에 목표·범위·보호·검증·미검증·다음 게이트를 기록한다.
- Actions 파일 존재, 실행 성공, Required Check 강제를 독립 상태로 관리한다.

## 기본 흐름

```text
사용자 요청
→ 저장소 사실·현재 상태 확인
→ 분야·최소 스킬 라우팅
→ 필요 시 딥인터뷰·마지막 확인
→ Issue 또는 승인 직접 요청 계약
→ Plan·완료·검증·중단 기준
→ 승인 범위 구현
→ 자동·수동 검증
→ 책임 원본·Registry·Context·Handoff 갱신
→ Draft PR·검토
→ Integration·Completion
```

## 구현 인수 흐름

```text
문서 승인
→ Windows 작업본 Git 상태 확인
→ 원격·로컬 차이 감사
→ 실제 Godot 경로·테스트 확인
→ 보호 경로·저장 계약 확정
→ 구현 Plan 승인
→ 작은 단위 구현·검증
→ 문서·테스트·Handoff 동기화
```

## Base 환류 흐름

```text
프로젝트 관찰·검증
→ 프로젝트 고유 값 분리
→ 기존 Base 중복·충돌 확인
→ [수정제안서] 제안 전용 PR
→ 사용자 검토·구현 승인
→ 별도 Base 구현 PR
→ 프로젝트 학습 기록 갱신
```

사용자가 특정 Base 변경을 직접 승인한 경우에만 제안 단계를 생략할 수 있다.

## 파일 작업 안전

- 혼합 작업트리에서는 관련 파일만 stage한다.
- 기존 파일 삭제·이동은 보존 대조와 사용자 승인을 요구한다.
- 새 버전·final·latest 복제본을 만들지 않는다.
- 동일 책임의 Markdown·JSON 원본을 동시에 유지하지 않는다.
- PDF·DOCX를 독립 원본으로 수동 수정하지 않는다.
- 실패 시 기존 정상 산출물을 보존한다.

## 작업 종료 보고

- 실제 변경.
- 제안만 한 변경.
- 제거한 중복과 보존한 기존 파일.
- 실행한 검증·실패·미검증.
- Base에 제안한 공용 교훈과 프로젝트에 남긴 고유 값.
- 다음 작업자의 첫 행동과 선행 조건.
