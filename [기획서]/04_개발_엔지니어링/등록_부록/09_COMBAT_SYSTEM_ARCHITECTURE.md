# 전투 시스템 아키텍처

> 원칙: 문파 무공이 새 전투 절차를 만들지 않는다. 공용 행동·합·자원 절차를 먼저 해상하고 무기·성급·심법·진의는 허용된 수정 지점만 바꾼다. 내부 데이터와 플레이어 표현을 분리한다.

## 1. 기술·출시 기준

- 엔진: Godot 4.7.1 + GDScript.
- 목표: Steam Windows 데스크톱.
- 데모: 5전 예선 결승까지.
- 전체판: 10전 결승까지.
- 전투 규칙은 씬과 분리된 순수 GDScript 시뮬레이터로 실행한다.
- UI는 입력과 이벤트 표시를 담당하며 전투 결과를 직접 계산하지 않는다.
- 상대의 내부 등급·행운·수련 스타일과 플레이어용 이명·풍문 표현을 분리한다.

## 2. 시스템 층

| 층 | 책임 |
|---|---|
| 공용 행동 | 접근, 후퇴, 공격, 막기, 회피, 밀치기, 교체, 재장전, 명상 |
| 선택 묶음 | 두 행동 잠금·동시 공개·순차 해상 |
| 무기 규칙 | 사거리, 기본 합, 일반 피해, 비용, 장전 |
| 합 규칙 | 쌍방 공격 비교와 차이 피해 |
| 핵심무공 | 3·6·9성 기술과 10성 절초 |
| 심법 | 패시브·기믹·명상 강화와 10성 진의 |
| 회차 성장 | 성과 등급, 행운, 수련 포인트, 제약, 보상 |
| 상대 생성 | 등급, 행운, 수련 스타일, 실제 투자, AI 성향 |
| 정보 표현 | 이명, 풍문, 전적, 정탐 문장 |
| 대회 진행 | 데모 5전·전체판 10전과 저장 |

## 3. 전투 시뮬레이션

| 구성 요소 | 책임 |
|---|---|
| `CombatState` | 위치, 체력, 행동력, 기력, 내공, 장전, 상태, 장비, 기믹 |
| `ActionChoice` | 타이밍 하나의 공용 행동과 무공·장비 참조 |
| `ActionPair` | 한쪽 전투원의 1수·2수와 잠금 상태 |
| `PairLockCoordinator` | AI 사전 잠금, 플레이어 잠금, 동시 공개 |
| `TimingResolver` | 이동·방어·공격·합·상태 해상 |
| `ClashResolver` | 최종 합 수치와 차이 피해 계산 |
| `RoundResolver` | 5개 묶음, 10타이밍, 종료 회복·상태 처리 |
| `CombatModifierSet` | 비용·합·피해·사거리·상태·명상 수정자 |
| `AiPlanner` | 공개 정보와 성향만으로 행동 두 개 생성 |

전투 이벤트는 `pair_locked`, `pair_revealed`, `move`, `guard`, `evade`, `attack`, `clash`, `hit`, `push`, `status`, `reload`, `meditate`, `resource`, `invalid_action`으로 반환한다.

## 6. 공용 수정자 계약

허용:

- 행동력·기력·내공·체력 비용.
- 합 수치와 상대 합 참조 비율.
- 일반 피해와 합 차이 피해 경감.
- 사거리·이동량.
- 막기·회피·밀치기.
- 상태 부여·중첩·지속.
- 장전·회수·명상 회복.
- 다음 행동·같은 계통 행동 강화.
- 2수 묶음 내 연속 사용.
- 발동 횟수와 최대 중첩.

금지:

- 문파 전용 행동 타입이나 해상 순서.
- UI에서만 계산되는 전투 결과.
- AI가 플레이어 비공개 행동을 읽는 기능.
- 심법 3개 이상 보유.
- 같은 계열의 두 번째 10성.
- 플레이어 표현 문구가 내부 전투 수치를 수정하는 구조.

## 7. 타이밍 해상 순서

1. 양측 행동과 부착 장비 확인.
2. 행동 가능 여부와 비용 판정.
3. 준비·다음 행동 수정자 적용.
4. 이동 동시 처리.
5. 막기·회피·방패 적용.
6. 이동 후 공격·밀치기 성립 확인.
7. 쌍방 공격이면 `ClashResolver` 실행.
8. 단독 공격이면 일반 피해 실행.
9. 상태·장전·회수·명상·자원 반영.
10. 구조화 이벤트와 다음 상태 반환.

`ClashResolver`는 기본 합 → 무기 → 성급 → 심법·기믹 → 상대 참조 반격 → 최종 합 → 차이 피해의 원인 목록을 반환한다.

## 8. 성장 규칙

- 시작 핵심무공과 심법은 3성.
- 성급 `n` 상승 비용은 `n`포인트.
- 정산식은 `기본 5 + 등급 보너스 + 행운 앞면 수`.
- 등급 보너스는 C1/B3/A5/S7.
- A는 평균 성과다.
- 행운 결과표:
  - C: 0/0/5/95%.
  - B: 5/5/5/85%.
  - A: 20/5/20/55%.
  - S: 60/10/15/15%.
- 플레이어 결과는 앞면 수만 `+`로 표시한다.
- 핵심무공은 여러 개 보유 가능, 심법은 최대 2개.
- 핵심무공 10성 슬롯과 심법 10성 슬롯을 별도로 저장한다.
- 데모 목표는 A4 약 50%, A3+S1 약 60%, A3+B1 약 30%의 5전 전 절초 개방이다.

## 9. 상대 생성

1. 등장 경기와 단계로 성장 횟수를 계산한다.
2. 기준 등급을 A계열 60%, B계열 40%로 추첨한다.
3. 해당 등급의 `LuckOutcomeTable`에서 행운 0~3을 추첨한다.
4. 경기당 `5 + 등급 보너스 + 행운`을 누적해 총 수련 예산을 구한다.
5. 균형형·핵심무공 올인형·심법 연계형·다무공형 중 수련 스타일을 정한다.
6. `TrainingAllocator`가 4→5→6성처럼 실제 단계 비용을 순서대로 지불한다.
7. 남은 포인트는 다음 성급의 부분 수련으로 저장한다.
8. 실제 무공·심법·절초·진의를 기준으로 AI 행동 프로필을 만든다.
9. 내부 종합 강도를 많이 약함·약함·중간·강적·매우 강함의 5단계로 계산한다.
10. `DiegeticHintResolver`가 플레이어용 이명·풍문·전적·정탐 문장을 선택한다.

상대 생성 결과는 회차 최초 생성 시 고정한다. 재도전과 저장·불러오기에서 다시 굴리지 않는다.

## 10. 무림식 정보 계약

- 개발용 등급·강도·스타일 태그는 기본 UI에서 비공개다.
- 이명은 종합 강도와 대표 전법을 암시한다.
- 풍문과 최근 전적은 행운과 성장세를 암시한다.
- 무공 평가는 투자 스타일과 성급 경계를 암시한다.
- 정탐 단계가 높아지면 문장이 더 구체적이 되지만 내부 원시 수치표를 그대로 출력하지 않는다.
- 같은 내부 의미는 같은 의미 키와 어휘 계열을 사용한다.
- 문구 선택은 전투 계산과 분리되며 실제 프로필과 모순될 수 없다.

## 11. 제약 규칙

- 제약은 전투 시작 전에 선택하고 공용 수정자를 초기 상태에 적용한다.
- 상대 정보 표시 뒤 제약 선택 화면으로 이동한다.
- 삼수양보 같은 시간 조건은 `timing_index`로 표현한다.
- 문파·무공 ID 직접 금지보다 공용 자원·거리·행동 조건을 우선한다.
- 승리했을 때만 추가 보상을 정산한다.

## 12. 대회·저장

- 데모: 1~2전 내부전, 3~4전 예선, 5전 예선 결승.
- 전체판: 8전 8강, 9전 준결승, 10전 결승.
- 저장 데이터는 포맷 버전, 회차, 문파, 경기·단계, 데모 종료, 상대 풀, 상대 성장 프로필과 표현 키, 보유 무공·심법·성급·부분 포인트, 10성 슬롯, 장비, 명망, 재도전, 제약을 보관한다.
- headless 테스트는 위치·2수 잠금·합·상태·수련·행운·상대 생성·표현 매핑·저장을 검증한다.
- 새 문파와 무공은 공용 엔진 수정 없이 데이터와 허용 수정자로 추가할 수 있어야 한다.
