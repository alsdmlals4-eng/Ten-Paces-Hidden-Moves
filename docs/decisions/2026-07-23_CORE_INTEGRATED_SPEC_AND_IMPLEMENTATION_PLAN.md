## 1. 제품 정의

### 1.1 프로젝트 정체성

> 상대의 공개된 상태와 반복 습관을 읽고, 서로의 현재 계획을 모른 채 10칸 전장에 `3수 → 3수 → 4수`로 수를 걸어 거리·`[합]`·대응·중단으로 한 수를 파훼하고, 그 이유를 복기해 다음 계획을 바꾸는 1대1 무협 심리 전술 로그라이트.

### 1.2 시장 차별화

> 미래를 미리 보는 무협 게임이 아니라, 보이지 않는 상대의 수를 읽고 파훼하는 무협 결투.

### 1.3 플레이어 약속

> 어려운 실시간 콤보를 외워 이기는 것이 아니라, 상대의 습관을 읽고 한 수 앞에서 파훼한 무인이 된다.

### 1.4 코어 우선순위

```text
대전 격투식 수읽기·파훼
> 전술 퍼즐식 거리·순서
> 로그라이트 성장
```

로그라이트 성장은 수읽기를 강화하는 보조 구조이며, 원시 능력치 상승으로 실패를 무력화하지 않는다.

## 2. 대상 플레이어와 플레이 조건

### 2.1 대상 플레이어

- 격투 게임의 심리전은 좋아하지만 실시간 입력·콤보 암기 부담은 줄이고 싶은 플레이어.
- 적은 규칙을 반복해서 익히며 거리·순서·대응 숙련을 높이는 전술 플레이어.
- 실패 원인을 이해하고 다음 시도에서 계획을 바꾸는 학습형 플레이어.
- 싱글플레이에서 읽을 수 있는 라이벌과 겨루고 싶은 플레이어.

### 2.2 플레이 조건

- 1대1 싱글플레이 결투.
- 한 화면에서 읽을 수 있는 10칸 일자형 전장.
- 현재 묶음 행동은 서로 비공개.
- AI는 플레이어의 미확정 계획을 읽지 않는다.
- 기초 행동은 항상 사용할 수 있는 소수 공용 어휘다.
- 판정은 결정론적이며 결과 이유를 설명할 수 있다.
- 포인터 외 키보드와 대체 정보 채널을 지원한다.

## 3. 코어 경험과 루프

### 3.1 뾰족한 재미

> 간파한 위험에 수를 걸고, 공개 순간 거리·합·대응이 맞아 상대 계획을 끊은 뒤, 그 근거를 복기해 다음 수에서 더 정확해지는 쾌감.

### 3.2 감정 흐름

```text
간파 → 결단 → 파훼 → 납득 → 적응
```

### 3.3 핵심 루프

| 단계 | 플레이어 행동 | 핵심 질문 | 필요한 피드백 |
|---|---|---|---|
| READ | 위치·자원·상태·행동 이력을 읽음 | 상대가 무엇을 노리는가 | 위험 거리·상태·이력 |
| HYPOTHESIZE | 상대 성향과 후보 수를 좁힘 | 가장 가능성 높은 의도는 무엇인가 | 공개 단서·반복 패턴 |
| PLAN | 행동·대상·자원을 3/3/4에 배치 | 어떤 순서가 여러 상대 수를 견디는가 | 대상·자원·준비/실행 미리보기 |
| COMMIT | 계획을 확정 | 이 위험에 걸 것인가 | 입력 잠금·상대 계획 공개 |
| RESOLVE | 대응→속공→이동→일반 공격 관찰 | 어느 교차점이 승부를 갈랐는가 | 합·방어·회피·중단·기세 |
| REVIEW | 예상과 실제 비교 | 무엇을 잘못 읽었는가 | 결정적 원인 요약 |
| ADAPT | 다음 계획 변경 | 다음에는 무엇을 바꿀 것인가 | 새 가설·재도전 |

### 3.4 보상 구조

- `Micro`: 합 승리, 정확한 대응, 중단, 거리 선점, 기세 획득.
- `Session`: 한 라이벌의 성향을 발견하고 결투 승리.
- `Meta`: 원시 능력치가 아니라 새로운 전투 관점·행동 변형·상대 정보 접근을 해금.

## 5. 현행 T0 통합 전투 명세

세부 예외와 수치는 `docs/02_COMBAT_RULES.md`가 책임진다.

### 5.1 전장

- 1~10번 일자형 10칸.
- 기본 시작: 플레이어 4번, 상대 7번.
- 시작 거리 3.
- 같은 칸 최대 2인, 거리 0 `[밀착]`.
- 자리 교환·상대 통과·전장 밖 이동 금지.

### 5.2 라운드와 계획

- 한 라운드 총 10수.
- 묶음은 3수 → 3수 → 4수.
- 현재 묶음만 편집한다.
- 진행 전까지 행동 전체 해제 가능.
- 진행 뒤 수정·환불 불가.

### 5.3 판정 순서

```text
대응 → 속공 → 이동 → 일반 공격
```

### 5.4 행동 어휘

- 이동.
- 보법.
- 막기.
- 회피.
- 속공.
- 강공.
- 명상.
- 태세.

### 5.5 핵심 판정

- 같은 실행 수의 유효 공격은 `[합]`.
- 합은 원공격력 차이를 정산한다.
- 방어도 차감 뒤 같은 수 막기는 남은 피해를 반감한다.
- 회피는 같은 수 유효 공격을 회피한다.
- 파공검기 `[필중]`은 회피를 무시한다.
- 피격 시 같은 수 미실행 행동을 중단한다.
- 태세 기반 `[강건]`은 제한된 중단 보호를 제공한다.

### 5.6 자원

- 체력.
- 기력.
- 내력.
- 기세.
- 비용은 행동 슬롯·기력·내력·기세다.
- 덱·손패·행동력·내공은 사용하지 않는다.

### 5.7 AI

- 입력은 공개 상태·이전 행동·재현 가능한 seed로 제한한다.
- 미확정 플레이어 슬롯·대상·절초 예약을 읽지 않는다.
- 현행 최소 AI는 기술 검증용이며 장기 라이벌 성향 시스템이 아니다.

### 5.8 종료와 재시작

- 승리·패배·무승부를 판정한다.
- 재시작은 4/7 위치·체력·자원·기세·상태·라운드·AI 이력·UI를 초기화한다.

## 7. 구현 원칙

### 7.1 판정 엔진 불변

- 복기 UI와 라이벌 성향은 기존 전투 결과를 다시 계산하지 않는다.
- `CombatResolutionEngine`이 반환한 `timing_results`와 `presentation_events`를 소비한다.
- 합·방어·필중·중단 공식 변경은 별도 수치 Issue와 플레이 근거가 있을 때만 수행한다.

### 7.2 AI 공정성

- AI 후보 생성 함수의 입력 스키마를 명시한다.
- 비공개 플레이어 계획 값이 입력·로그·debug snapshot에 들어오면 테스트를 실패시킨다.
- 무작위는 후보 선택에만 제한적으로 사용하며 seed로 재현 가능해야 한다.

### 7.3 데이터 우선

- 라이벌 성향·공개 단서·후보 가중치는 JSON 데이터가 소유한다.
- GDScript에 특정 라이벌 이름·수치를 하드코딩하지 않는다.
- 새 상대가 동일 스키마를 재사용할 수 있어야 한다.

### 7.4 접근성

- 색·모션·음향 하나만으로 결과를 전달하지 않는다.
- 결정적 복기 문장은 텍스트와 포커스 가능한 UI로 제공한다.
- 모션 감소에서는 연출을 생략해도 판정 순서와 결과가 보존돼야 한다.

### 7.5 범위 통제

- 코어 검증 슬라이스에서는 새 무공·세력·경제·캠페인 시스템을 추가하지 않는다.
- 기존 행동 수를 늘리지 않는다.
- 먼저 플레이어가 현재 8개 행동 관계를 이해하는지 검증한다.

## WP1. 결정적 복기 MVP

### 목적

플레이어가 상세 로그를 읽기 전에 자신의 가설이 어디서 맞고 틀렸는지 이해하게 한다.

### 사용자 흐름

```text
묶음 확정
→ 상대 계획 공개
→ 수별 판정 재생
→ 결정적 복기 카드
→ 다음 묶음 편집 또는 재시작
```

### 필수 정보

- 내 계획 요약.
- 상대 실제 계획 요약.
- 시작·종료 위치와 거리.
- 승부를 가른 첫 결정적 수.
- 결정적 원인: 거리 / 방향 / 합 / 방어 / 회피 / 필중 / 중단 / 자원.
- 다음 계획에서 바꿀 수 있는 선택 하나.

### 비목표

- AI가 정답 행동을 추천하지 않는다.
- 모든 수를 장문으로 설명하지 않는다.
- 판정 엔진을 수정하지 않는다.
- 승률·정답률을 표시하지 않는다.

### 예상 파일

- `src/ui/`에 결정적 복기 패널 또는 기존 로그 패널 확장.
- `scenes/ui/` 또는 현행 전투 씬의 복기 영역.
- `src/combat/`에는 읽기 전용 summary builder를 둘 수 있으나 판정 변경 금지.
- `data/ui/`가 존재하면 복기 라벨·우선순위 데이터.
- `tests/`에 summary contract와 UI focus 테스트.

### 데이터 계약 후보

```yaml
review_summary:
  player_plan:
  enemy_plan:
  decisive_timing:
  cause_code:
  cause_label:
  state_before:
  state_after:
  suggested_dimension:
```

`suggested_dimension`은 정답 행동이 아니라 `거리`, `대응`, `자원`, `순서` 같은 검토 차원만 제공한다.

### 수용 기준

- 같은 판정 입력은 같은 복기 요약을 만든다.
- 합·방어·회피·필중·중단 각각의 대표 fixture가 올바른 원인을 선택한다.
- 모션 감소에서도 동일 텍스트를 읽을 수 있다.
- 키보드로 복기 → 상세 로그 → 다음 계획 이동이 가능하다.
- 실제 플레이어가 결정적 원인 하나를 자기 말로 설명할 수 있는지 STEP 14에서 측정한다.

## WP2. 읽을 수 있는 라이벌 성향 MVP

### 목적

AI를 고정 퍼즐이나 근거 없는 무작위가 아니라 관찰 가능한 습관을 가진 라이벌로 만든다.

### 라이벌 최소 데이터

```yaml
rival_id:
display_name:
archetype:
public_clues:
  - clue_id:
    text:
tendencies:
  preferred_distance:
  aggression:
  guard_bias:
  evade_bias:
  resource_conservation:
  ultimate_risk:
candidate_policy:
  max_candidates:
  deterministic_seed_scope:
  tie_breaker:
counter_windows:
  - condition:
    readable_hint:
```

### 후보 생성 원칙

1. 공개 상태에서 유효 행동 후보를 생성한다.
2. 라이벌 성향으로 후보 점수를 조정한다.
3. 최상위 복수 후보 중 seed 기반으로 선택한다.
4. 선택 근거를 debug trace에 기록한다.
5. 플레이어의 미확정 계획은 입력하지 않는다.

### 라이벌 1호 실험

T0에서는 하나의 라이벌만 구현한다.

- 중거리 선호.
- 거리가 맞으면 강공 준비 확률 증가.
- 피해가 예상되면 막기 편향.
- 기세 5일 때 파공검기 선호.
- 낮은 자원에서는 명상·거리 이탈 편향.

위 항목은 수치가 아니라 성향 문법 검증용이다.

### 비목표

- 머신러닝 AI.
- 플레이어 행동을 비공개로 예측하는 모델.
- 난이도별 별도 판정 규칙.
- 상대마다 독립 GDScript 구현.

### 예상 파일

- `data/combat/` 또는 `data/rivals/` 라이벌 성향 JSON.
- `src/combat/combat_ai_planner.gd` 후보 생성·점수·seed 선택 분리.
- `src/combat/`에 policy/trace helper.
- 전투 전 또는 HUD의 공개 성향 단서 UI.
- `tests/`에 입력 차단·seed 재현·성향 분포 테스트.

### 수용 기준

- 동일 공개 상태·동일 seed에서 동일 결과.
- seed 변경 시에도 정의된 합리적 후보 집합 밖으로 나가지 않음.
- 미확정 플레이어 계획 값을 전달하면 테스트 실패.
- 최소 20회 시뮬레이션에서 성향이 선택 분포에 드러남.
- 신규 플레이어 5명 중 3명 이상이 안내 없이 성향 하나를 발견하는지 STEP 14에서 측정.

## WP3. 단계적 온보딩과 정보 위계

### 목적

8개 행동을 삭제하지 않고 관계를 순차적으로 이해하게 한다.

### 단계

| 단계 | 전면 노출 | 접는 요소 | 통과 조건 |
|---|---|---|---|
| A | 이동·속공·막기·3수 계획 | 보법·회피·강공·명상·태세·절초 | 위치·공격·막기 이해 |
| B | 보법·회피·강공 | 명상·태세·절초 | 거리·방향·대응 차이 이해 |
| C | 명상·태세·기세 | 절초 세부 | 자원·중단·강건 이해 |
| D | 절초 3종 | 없음 | 역할 차이와 기세 결단 이해 |

### 원칙

- 실제 기능을 삭제하지 않고 안내·강조·노출 순서만 조정한다.
- 첫 플레이에서 모든 수치 설명을 강요하지 않는다.
- 기본 화면은 거리·상태·현재 묶음·결정적 위험을 우선한다.
- 상세 수치는 펼침 영역에서 확인한다.

### 수용 기준

- 첫 계획까지 필요한 핵심 정보가 한 화면에서 보인다.
- 도움말 없이 행동 이름과 대상 선택 흐름을 완료한다.
- 키보드 포커스 순서가 시각 순서와 일치한다.
- 접근성 옵션으로 모션·음향·색 의존을 대체한다.

## WP5. STEP 14 발견형 플레이테스트

### 표본

- 신규 플레이어 5명.
- 전술·격투 관심은 있으나 현행 빌드 사전 지식이 없는 참가자 우선.
- 동일 빌드와 동일 기본 안내.
- 안내 결투 1회 뒤 자율 재도전 1회.

### 관찰 과제

1. 첫 묶음을 스스로 계획한다.
2. 결과 뒤 가장 중요한 원인을 설명한다.
3. 상대의 반복 성향을 하나 추정한다.
4. 재도전에서 계획을 바꾼다.
5. 접근성 장벽을 보고한다.

### 발견형 통과 신호

- 5명 중 4명 이상이 치명적 차단 없이 한 판 완료.
- 5명 중 4명 이상이 3/3/4와 결정적 원인 하나를 설명.
- 5명 중 3명 이상이 안내 없이 상대 성향 하나를 발견.
- 5명 중 3명 이상이 재도전에서 계획을 실질적으로 변경.
- 5명 중 3명 이상이 자발적으로 다시 하거나 구체적 다음 수를 제시.
- 핵심 정보가 색·모션·음향 하나에만 의존한 참가자가 없음.

### 실패 처리

- 성공 기준을 사후 변경하지 않는다.
- 실패 원인을 규칙·UI·AI·온보딩·성능·접근성으로 분류한다.
- `KEEP / AMPLIFY / CHANGE / REMOVE / DEFER / RETEST`를 기록한다.
- T1을 시작하지 않고 `REPEAT_POC`한다.

## 14. 롤백과 중단 조건

### 롤백

- 각 WP를 독립 브랜치와 PR로 유지한다.
- 데이터 schema 변경 시 이전 fixture를 보존하거나 명시적 migration을 둔다.
- 복기 UI는 feature flag 또는 독립 scene 경계로 제거 가능하게 한다.
- 라이벌 성향 실패 시 현행 최소 AI로 되돌릴 수 있게 입력·출력 인터페이스를 유지한다.

### 중단·축소

- 상대 읽기보다 고정 알고리즘 암기가 우세하다.
- 결과 요약 뒤에도 실패 이유를 설명하지 못한다.
- 3/3/4가 결단이 아니라 입력 대기열로 느껴진다.
- 합·절초·기세 하나가 대부분의 선택을 지배한다.
- 코어보다 콘텐츠 수·성장 수치·대회 분량이 우선된다.
- 새 상대·무공마다 독립 판정 절차가 필요하다.
- 접근성·성능·제작 비용이 목표 범위를 초과한다.

중단 조건이 나타나면 T1으로 넘어가지 않고 범위·규칙·정보 표현을 축소한다.
