# Godot 2D 12세력 대회 세로 슬라이스 설계

## 1. 결정

- 엔진은 **Godot 4.7.1 + GDScript**를 사용한다.
- 출시 목표는 Steam의 Windows 데스크톱 빌드다. Steamworks 연동과 상점·업적은 이번 세로 슬라이스 범위 밖이다.
- 전투는 2D 가로 경기장이다. 10칸의 위치, 6슬롯 행동열, 자원과 결과 로그를 캐릭터 연출보다 우선해 보여 준다.
- 전투 규칙은 화면과 독립된 순수 GDScript 시뮬레이터로 실행한다. UI는 입력을 수집하고 결과를 재생만 한다.
- 이번 구현은 시작 6세력과 추가 6세력, 내부전부터 결승까지 9전 회차를 포함한다.

Godot 4.7.1은 설계 시점의 안정 릴리스다. 개발 환경은 해당 버전과 동일한 메이저·마이너 버전을 고정한다.

## 2. 사용자 흐름

1. 플레이어는 화산·개방·당문·소림·무당·마교 중 시작 문파를 고른다.
2. 대회 화면에서 현재 단계, 다음 상대의 공개 정보, 보유 무공·대성·장비를 확인한다.
3. 전투 화면에서 A=4, B=6으로 시작한 10칸 경기장과 6개 행동 슬롯을 본다.
4. 플레이어는 행동열을 전부 잠근다. AI는 이미 공개 정보만으로 행동열을 잠근 상태다.
5. 양측 행동열을 동시에 공개하고, 슬롯별로 이동·방어·공격·상태를 해상한다.
6. 경기 뒤 무공서·수련도·보조장비·회복·정탐 중 하나를 고른다.
7. 9전까지 진행하며, 패배 시 회차당 한 번 재도전할 수 있다.

## 3. 아키텍처

### 3.1 시뮬레이션 계층

`combat/`은 Godot 노드, 씬, 애니메이션에 의존하지 않는다. 입력 상태와 불변 데이터에서 이벤트 목록과 다음 상태를 반환한다.

| 구성 요소 | 책임 |
|---|---|
| `CombatState` | 위치, 체력, 행동력, 기력, 내공, 장전, 상태, 장비, 대성 상태 보관 |
| `ActionOrder` | 6슬롯 행동과 장비 부착 행동을 검증 가능한 형태로 보관 |
| `CombatResolver` | 슬롯 하나를 해상하고 전투 이벤트·다음 상태 반환 |
| `RoundResolver` | 6개 슬롯과 라운드 종료 회복·독·마화를 순서대로 실행 |
| `FactionModifier` | 유파·대성 효과가 수정할 수 있는 수치와 조건을 제공 |
| `AiPlanner` | AI의 공개 정보와 성향만 받아 행동열 생성 |

전투 이벤트는 UI가 직접 추론하지 않도록 `move`, `guard`, `evade`, `hit`, `push`, `status`, `reload`, `resource`의 구조화된 기록으로 반환한다. 결과 로그는 이벤트를 한국어·중국어·영어 문자열로 변환하는 별도 표시 계층에서 만든다.

### 3.2 데이터 계층

문파·무기·무공·보조장비·AI 성향은 `.tres` Resource 또는 JSON이 아니라 Godot Resource로 저장한다. 에디터에서 조정할 수 있고, 시뮬레이터는 데이터 ID만 참조한다.

| 데이터 | 최소 필드 |
|---|---|
| `WeaponData` | ID, 사거리, 피해, 행동력·기력 비용, 막기·밀치기 효율, 재장전 규칙 |
| `FactionData` | ID, 분류, 기본 무기, 시작 무공, AI 성향, 대표 거리 |
| `MartialArtData` | 문파 ID, 1~5성 효과, 허용 수정자, 해금 수련도 |
| `AccessoryData` | 호환 무기, 사용 횟수, 부착 대상 행동, 비용 면제 효과 |
| `OpponentProfile` | 세력, 난이도 단계, 보유 무공·장비, AI 성향 |
| `TournamentStageData` | 경기 번호, 단계, 상대 풀, 보상 수, 정탐 가능 여부 |

무공은 새 행동 타입을 만들지 않는다. 오직 사거리, 비용, 피해, 경감, 이동량, 상태 부여, 재장전, 사용 횟수라는 수정자를 통해 공용 행동을 바꾼다.

### 3.3 화면 계층

| 씬 | 책임 |
|---|---|
| `MainMenu` | 새 회차 시작과 저장 회차 선택 |
| `FactionSelect` | 시작 6세력의 무기·대표 규칙·추천 거리 표시 |
| `TournamentScreen` | 9전 진행, 다음 상대, 보상, 대성, 재도전 진입 |
| `BattleScreen` | 10칸 보드, 양측 상태, 6슬롯 행동 배치, 동시 공개, 이벤트 재생 |
| `ResultScreen` | 승패, 보상, 수련도, 재도전 선택 |

`BattleScreen`은 전투 규칙을 계산하지 않는다. 행동열을 `CombatState`와 함께 시뮬레이터에 넘기고, 반환된 이벤트를 순서대로 2D 스프라이트·이펙트·로그로 표현한다.

## 4. 전투 계약

- 보드는 1~10칸이며, 모든 경기 시작 위치는 A=4, B=6이다.
- 같은 칸 중첩, 교차, 자리 교환이 가능하다.
- 양측은 6개 시간 슬롯을 비공개로 모두 선택하고 동시 공개한다.
- 공용 행동은 접근, 후퇴, 공격, 막기, 회피, 밀치기, 무기 교체, 재장전이다.
- 슬롯 해상 순서는 행동 가능·비용 → 동시 이동 → 방어·회피 → 이동 후 사거리 공격·밀치기 → 피해·상태 → 라운드 종료 회복·상태다.
- AI는 플레이어의 숨은 행동열을 절대 읽지 못한다. 행동열 생성 뒤에는 바꿀 수 없다.
- 일반 막기는 방패 없이도 가능하다. 방패는 활 피해 경감을 높이고 회피 효율을 낮춘다.
- 암기벨트와 화살통은 제한 횟수 안에서 기존 행동에 부착돼 슬롯·행동력 비용을 면제한다.
- 마화는 체력을 소모해 강한 효과를 내고, 중첩은 라운드 종료 체력 감소를 만든다.

## 5. 12세력 세로 슬라이스

시작 문파는 화산·개방·당문·소림·무당·마교다. 추가 전투 세력은 양가장·백우궁·관군·아미·오독·녹림채다.

각 세력은 기본 무기, 대표 1성 규칙, 2~4성 강화, 5성 대성, AI 거리 성향을 하나씩 가진다. 녹림채는 도적·산적을 포괄한 사파 세력이며, 도 공격 적중 시 상대 기력을 빼앗는 약탈도를 대표 규칙으로 쓴다.

상세 수치와 효과는 `docs/06_STARTING_FACTION_MASTERY_DATA.md`의 데이터 표를 단일 기준으로 한다. 코드에는 문파 이름을 기준으로 한 전용 분기를 두지 않는다.

## 6. 9전 대회와 저장

| 경기 | 단계 | 시스템 검증 |
|---:|---|---|
| 1~2 | 내부전 | 선택 문파의 기본 무기·규칙 |
| 3 | 추천패 쟁탈전 | 첫 외문 무공 또는 보조장비 |
| 4~5 | 지역 예선 | 무기 상성과 보상 선택 |
| 6~7 | 본선 | 3~4성 수련과 정탐 |
| 8 | 준결승 | 대성 또는 혼합 빌드 |
| 9 | 결승 | 거리·자원·행동열 종합 |

저장 데이터는 회차 상태, 선택 문파, 현재 경기, 상대 풀, 보유 무공·수련도·장비·명망·재도전 사용 여부를 보관한다. 저장 포맷은 버전 필드를 포함하며, Steam Cloud 연동은 이후 Plan에서 다룬다.

## 7. 오류 처리와 검증

- 행동열 배치 시 비용 부족, 사거리 불가, 장전 필요, 장비 사용 횟수 부족을 즉시 거부하고 이유를 UI에 표시한다.
- 해상 중 유효하지 않은 행동이 발견되면 상태를 망가뜨리지 않고 `invalid_action` 이벤트를 남긴 뒤 해당 행동을 대기로 처리한다.
- 전투 시뮬레이터는 headless 테스트로 실행한다. Godot 씬 없이 거리·동시 이동·피해·독·마화·재장전·대성·AI 잠금을 테스트한다.
- 12세력·9전 회차는 통합 테스트에서 결승까지 진행돼야 한다.

## 8. 완료 정의

- 시작 6세력으로 새 회차를 시작할 수 있다.
- 추가 6세력이 대진 적·외문 무공서로 작동한다.
- 12세력의 대표 규칙과 1~5성 대성이 데이터에서 전투 결과로 반영된다.
- 9전 회차가 내부전부터 결승까지 저장·불러오기와 함께 진행된다.
- 결과 로그가 공용 행동 → 무기 → 유파·대성 → 최종 결과를 설명한다.
- Windows용 Godot 데스크톱 빌드를 만들 수 있다.

## 9. 명시적 비범위

- Steamworks 상점, 업적, 클라우드, 도전 과제
- 세계관 설명, 대화, 관계 시스템
- 무작위 손패·덱빌딩
- 별도 장비 강화·경제 시스템
- 멀티플레이와 PvP 밸런스
