## 1. 로드맵 원칙

- 가장 위험한 가설을 가장 작은 플레이 가능 범위에서 먼저 검증한다.
- 구현·정적 검사·Godot 런타임·Windows 기술 증거·사람 플레이를 독립 상태로 기록한다.
- T0 전투 코어가 사람 증거를 통과하기 전 세력·무공·성장 콘텐츠를 대량 제작하지 않는다.
- 사용자 승인 규칙이 바뀌면 정본·데이터·fallback·코드·테스트·UI·Skill·Context를 같은 변경에서 정렬한다.
- 백업·보류·과거 Plan·Git 이력은 기록으로 보존하되 활성 구현의 기본 입력으로 사용하지 않는다.
- 각 단계는 결과에 따라 `KEEP / AMPLIFY / CHANGE / REMOVE / DEFER / RETEST`한다.

## 5. 다음 작업 — 프로젝트 코어 확정

PLAN 모드에서 다음 순서로 진행한다.

1. 계속 플레이하게 만드는 핵심 컨셉 후보를 찾는다.
2. 프로젝트의 제약·조건·미학·제작 현실을 확인한다.
3. 뾰족한 재미를 하나의 플레이어 변화로 정의한다.
4. 모든 요소를 WHY/HOW/WHAT으로 대조한다.
5. 현재 POC가 해당 재미를 실제로 검증하는지 판정한다.
6. 불필요한 요소를 제거·보류하고 부족한 최소 실험을 정한다.
7. SWOT·VRIO로 프로덕션 경쟁력과 조직 가능성을 검토한다.
8. 사용자 승인으로 프로젝트 코어를 확정한다.

코어 확정 전에는 현재 구현을 암묵적 최종 코어로 취급하지 않는다.

## 7. T1 — 최소 세로 슬라이스

T0 사람 증거와 프로젝트 코어 승인을 모두 통과한 뒤 시작한다.

- 플레이 스타일 2개.
- 상대 3명 내외.
- 전투 3회 내외.
- 전투 전 정보.
- 전투 후 보상 선택.
- 강화 1회 이상.
- 최종 강적.

목적은 5전 데모 제작 전에 `정보 → 전투 → 보상 → 성장 → 다음 전투`가 반복 동기를 만드는지 증명하는 것이다.

## 10. 공통 검증 게이트

각 단계 종료 시 다음 순서로 확인한다.

```text
계약·범위
→ 정본·경로·ID·Schema 최신성
→ 형식·문법·정적 검사
→ 자동 테스트
→ Godot 런타임·렌더·빌드
→ 적용 시 접근성 사용자 검수
→ 적용 시 목표 플랫폼 성능
→ 정상·실패·경계·반례·회귀
→ 증거 보고
→ Context·Roadmap·Handoff·PR 갱신
```

실행하지 않은 검증은 `NOT_RUN` 또는 `UNVERIFIED`다.

## 11. 중단·축소 조건

- 코어보다 콘텐츠 수가 우선되기 시작한다.
- 규칙 실패를 플레이어가 설명하지 못한다.
- 상대를 읽는 재미보다 AI 치팅 의심이 강하다.
- 3/3/4가 의미 없는 대기열로 느껴진다.
- 새 콘텐츠마다 독립 판정 절차가 필요하다.
- 접근성·성능·제작 비용이 목표 범위를 초과한다.

이 경우 다음 단계로 넘어가지 않고 범위·규칙·정보 표현을 축소·재조정한다.
