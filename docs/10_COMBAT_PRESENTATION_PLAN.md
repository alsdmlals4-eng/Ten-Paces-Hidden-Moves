# 십보강호 전투 연출 기획서

> 책임: 현재 T0 전투의 시선 흐름·계획·즉시 판정 피드백과 향후 단계별 무협 연출 계약  
> 규칙 원본: `docs/02_COMBAT_RULES.md`  
> UI 원본: `docs/07_COMBAT_UI_SPEC.md`  
> 범위·검증: `docs/05_COMBAT_POC_SPEC.md`, `docs/08_TEST_CHECKLIST.md`

## 1. 상태 구분

- `CURRENT`: 실제 POC가 지금 표현하는 방식.
- `IMPLEMENTED_FOR_REVIEW`: 코드에는 있으나 최신 사용자 확인 대기.
- `PLANNED`: 후속 STEP에서 구현.
- `PRESENTATION_TARGET`: T0 플레이테스트 뒤 필요성이 확인되면 제작.

현재 구현과 목표 연출을 같은 문장으로 설명하지 않는다.

## 2. 연출 목표

십보강호의 전투 연출은 화려함보다 다음을 우선한다.

1. 지금 무엇을 선택해야 하는지 이해한다.
2. 플레이어 4번·상대 7번의 시작 거리 3을 즉시 읽는다.
3. 배치한 수와 실제 판정의 관계를 신뢰한다.
4. 거리·방향·자원·대응·실패 원인을 다음 계획에 사용한다.
5. 계산을 방해하지 않으면서 무협 공방의 손맛을 제공한다.
6. 자산·모션·음향이 없어도 같은 결과를 읽는다.
7. 같은 단계 동시 공격과 이후 행동 중단을 시각 순서 때문에 오해하지 않는다.

## 3. 현재 T0 전투 흐름

```text
플레이어 4번·상대 7번과 시작 거리 3 확인
→ 양측 상태 확인
→ 현재 묶음 행동 배치
→ 이동 목적지·공격 방향 지정
→ 예상 기력·내력 확인
→ 진행
→ 대응 → 속공 → 이동 → 일반 공격 동기 판정
→ 확정 HUD·위치·슬롯·문자열 로그 즉시 반영
→ 다음 묶음 또는 다음 라운드
```

현재 특징:

- `[진행]` 신호 처리 안에서 한 묶음 판정을 즉시 완료한다.
- 판정 단계 사이의 시간차 애니메이션은 없다.
- 캐릭터 위치와 HUD는 최종 결과로 바로 갱신된다.
- 세부 판정 순서는 전투 기록에서 확인한다.
- 현재 POC에는 별도 합 수치 비교·차이 피해가 없다.

## 4. 상태 소유 경계

| 계층 | 책임 | 금지 |
|---|---|---|
| 전투 도메인 | 위치·비용·대응·피해·자원·라운드 | 모션 완료 여부로 판정 변경 |
| 판정 반환 | 확정 state·해결 행동·방어 profile·로그 | 장면용 임의 피해 추가 |
| UI | 입력·선택 가능 상태·예상 자원·확정 결과 표현 | 피해·회복·승패 재계산 |
| VFX·애니메이션·오디오 | 확정 결과 재생 | 보상·저장·기세 직접 지급 |

연출 콜백에서 체력·기력·내력·기세를 바꾸지 않는다.

## 5. 첫 시선

전투 진입 뒤 권장 시선 순서:

```text
중앙 10칸 전장
→ 플레이어 4번·상대 7번과 사이 빈 칸
→ 상단 체력·기력·내력·절초 기세
→ 라운드·행동 묶음·판정 순서
→ 하단 현재 행동 슬롯
→ 기초 행동 8종
```

- 전장과 캐릭터가 카드 상세·로그보다 먼저 읽혀야 한다.
- 시작 거리 3은 별도 숫자 팝업이 없어도 타일 번호와 위치로 읽을 수 있어야 한다.
- 현재 편집 수와 미래 잠금 수를 즉시 구분한다.
- 상세와 기록은 필요할 때만 펼치는 오버레이다.
- 수묵 배경이 전장 칸·캐릭터·HUD 대비를 방해하지 않아야 한다.

## 6. 계획 단계 표현

### 6.1 카드 선택

CURRENT:

- 호버하면 상세 임시 보기.
- 클릭하면 상세 고정과 행동 선택.
- 같은 카드 재클릭 또는 빈 전장 클릭으로 해제.
- 태세·대응 조합 대기 상태.

PRESENTATION_TARGET:

- 상세 고정과 행동 선택을 별도 아이콘·테두리로 더 명확히 구분.
- 키보드 포커스와 선택 상태를 동일하게 표현.

### 6.2 슬롯 배치

- 빈 수에 행동 이름을 표시한다.
- 2슬롯 행동은 두 칸이 하나의 준비 행동임을 연결한다.
- 마지막 점유 수가 실제 실행 수임을 상세 또는 표식으로 전달한다.
- 묶음 경계 초과·점유 충돌은 배치하지 않고 원인을 로그에 기록한다.
- 점유 칸을 누르면 행동 전체를 해제한다.

### 6.3 대상 지정

시작 위치 기준 이동:

```text
이동 배치
→ 4번에서 3번·5번 녹색 표시
→ 도착 칸 선택
→ 슬롯에 방향·칸 번호
```

시작 위치 기준 보법:

```text
보법 배치
→ 4번에서 2번·3번·5번·6번 후보
→ 전장 밖·상대 점유 칸 제외
→ 선택 목적지 표시
```

공격:

```text
공격 배치
→ 붉은 좌·우 방향 후보
→ 방향 선택
→ 슬롯에 공격 방향
```

- 색뿐 아니라 테두리·문구·방향 표식을 함께 사용한다.
- 공격 후보는 실제 피격 칸이 아니라 방향 선택임을 오해하지 않게 한다.
- 시작 거리 3에서는 속공·강공이 당장 적중하지 않는다는 점을 결과 로그가 설명한다.
- 지정이 끝나면 다음 미지정 행동으로 이어질 수 있다.

### 6.4 자원 예상

IMPLEMENTED_FOR_REVIEW:

- 배치 즉시 기력·내력 예상값 갱신.
- 실행 순서에 따른 명상 회복.
- 부족 슬롯 문구와 진행 잠금.
- 행동 해제 시 재계산.

수용 기준:

- 예상값을 실제 확정값과 혼동하지 않는다.
- 판정 뒤 실제 값이 직전 예상과 일치한다.
- 부족 원인이 어느 행동인지 확인한다.

## 7. 진행 버튼·현재 판정 전환

CURRENT 상태 문구:

| 상태 | 표시 |
|---|---|
| 조건 미충족 | `행동·대상·자원 확인 필요` |
| 조건 충족 | `배치 완료` |
| 동기 판정 완료 | `판정 완료` |

현재는 별도 `판정 중` 화면이 없다.

```text
진행 클릭
→ 동기 판정
→ 다음 묶음 상태와 판정 완료 문구
```

PRESENTATION_TARGET:

단계별 애니메이션을 도입할 때만 다음 상태를 추가한다.

```text
committed
→ resolving
→ presenting_result
→ next_bundle_ready
```

- `resolving` 동안 슬롯·카드·대상 입력을 명시적으로 잠근다.
- 스킵·빠른 재생을 제공해 반복 전투의 대기 시간을 줄인다.
- 애니메이션을 끄면 같은 상태 전환을 즉시 완료한다.

## 8. 판정 표현 순서

규칙과 같은 순서를 사용한다.

```text
대응 준비
→ 속공
→ 이동
→ 일반 공격·일반 행동
→ 피해·자원·기세 결과
```

### 8.1 CURRENT

- 단계별 애니메이션 대신 최종 위치·HUD와 문자열 로그를 즉시 표시한다.
- 같은 단계 공격은 판정 엔진에서 동시 처리된다.
- UI 표시 순서가 판정 순서를 바꾸지 않는다.

### 8.2 4/7 첫 묶음 기준 연출

통합 fixture 예:

```text
초기: 플레이어 4 / 상대 7
1수 이동 결과: 플레이어 5 / 상대 6
2수: 명상과 막기
3수: 인접 상태에서 양측 속공
```

- 이동 전과 이동 후 위치를 로그에서 구분한다.
- 플레이어와 상대가 같은 수에 이동하면 양측 결과를 같은 단계로 묶는다.
- 3수 공격이 적중하는 이유는 `이동 후 인접`으로 설명한다.

### 8.3 PRESENTATION_TARGET

대응:

- 막기: 자세·푸른 표식·감소량.
- 회피: 잔상 또는 명확한 회피 표식.
- 태세+막기: 묶음 지속 강화 방어 표식.
- 태세+회피: 묶음 지속 회피 표식.

속공:

- 이동 전에 공격 방향과 실행 수를 강조.
- 적중·방향 실패·사거리 실패를 구분.

이동:

- 출발·도착 칸을 먼저 강조.
- 실패 이동을 실행했다가 되돌리는 연출 금지.
- 동일 목적지 충돌은 양측 표식과 원인 문구 제공.

일반 공격:

- 이동 후 실제 거리와 방향을 표현.
- 같은 단계 쌍방 공격을 한쪽 선공으로 취소된 것처럼 연출하지 않음.
- 양쪽 피해를 동시에 또는 같은 판정 묶음으로 표시.

일반 행동:

- 명상 회복과 단독 태세 저장을 공격과 구분.

## 9. 한 수의 결과 피드백

한 수는 다음 질문에 답해야 한다.

- 누가 어떤 행동을 했는가?
- 어느 판정 단계인가?
- 출발·도착 위치는 어디인가?
- 방향과 거리가 유효한가?
- 회피·막기·태세 중 무엇이 적용됐는가?
- 원피해·감소량·최종 피해는 얼마인가?
- 위치·기력·내력·기세가 어떻게 변했는가?
- STEP 11 이후 어떤 행동이 왜 중단됐는가?

권장 로그:

```text
[1라운드 1묶음 판정]
[1수] 강호낭인이 4번에서 5번으로 이동했다.
상대는 7번에서 6번으로 이동했다.
[3수] 강호낭인이 오른쪽으로 속공했다.
거리 1에서 적중했다.
```

피할 표현:

```text
priority=20 / emitted=true / damage_result=2
```

개발 진단은 별도 로그나 디버그 snapshot에서 확인한다.

## 10. 전투 기록 패널

CURRENT:

- 기본 접힘.
- `기록` 버튼으로 펼침·접음.
- 문자열 로그 추가·초기화.
- 사용자가 직접 스크롤.

현재 없는 기능:

- 새 로그 후 자동 최하단 이동.
- 사용자가 과거 로그를 읽을 때 자동 이동 일시 정지.
- 로그 필터·검색·수별 접기.
- 구조화 event ID와 현지화.

PRESENTATION_TARGET:

- 라운드→묶음→수 계층.
- 계획 로그와 판정 로그 구분.
- 최신 로그 자동 이동과 사용자 스크롤 보호.
- 한 수 요약과 상세 수치 펼치기.

향후 읽기 성공 표식 후보:

- 간파: 정확한 대응.
- 파훼: 준비 행동 중단.
- 선점: 이동 후 위치 예측 공격.
- 받아냄: 큰 피해를 막기로 감소.
- 허초: 상대 대응을 낭비시킴.

표식은 현재 구현이 아니며 STEP 14에서 필요성을 검증한다.

## 11. 카드·상세 연출

카드 본체:

- 소속 `[기초]`.
- 이름·기능 배지·사거리.
- 슬롯·기력·내력.
- 원화 또는 SVG placeholder.

상세 패널:

- 대상·방향.
- 피해·이동량·방어도·회복.
- 조건·태그.
- 전체 효과와 대응 결합.

카드 본체·호버·클릭 상세에 같은 전문을 세 번 반복하지 않는다.

## 12. STEP 11 중단 연출 예정

현재 `PLANNED`다.

필요 흐름:

1. 준비 중 행동과 실행 예정 수.
2. 실제 피해 발생.
3. 이후 미실행 중단 대상 표시.
4. 점유 슬롯 전체 취소 표식.
5. 중단 원인.
6. 집중·강건에 의한 방지.
7. 사망·강제 제어 우선 처리.

중단 연출은 이미 같은 단계에서 성립한 공격을 취소한 것처럼 보이면 안 된다.

## 13. STEP 13 승패·재시작 예정

- 체력 0 뒤 추가 계획 입력 잠금.
- 승자·패자·종료 원인.
- 마지막 수 요약.
- 재시작.
- 재시작 시 플레이어 4번·상대 7번으로 복원.
- 전장·상태·라운드·슬롯·대상·로그 초기화.

전체 대회 결과·보상·수련은 T1 이후다.

## 14. 접근성 폴백

### 현재

- 이동·공격 후보는 색과 상태 이름을 함께 가진다.
- 자산이 없어도 기본 SVG·벡터 도형·텍스트로 화면이 구성된다.
- 주요 버튼은 포인터 입력 기준이며 키보드 전체 흐름은 아직 없다.

### ACCESSIBILITY_TARGET

- 색 없이 이동·공격·선택·잠금·부족 구분.
- 카드→슬롯→대상→진행→상세→기록 키보드 흐름.
- 항상 보이는 포커스.
- 점멸·흔들림·빠른 모션 감소.
- 모션을 꺼도 단계·결과 문구 유지.
- 음향이 없어도 텍스트·아이콘으로 동일 정보 제공.

접근성 사용자 검수는 `NOT_RUN`이다.

## 15. 성능 원칙

- 연출이 판정 실행이나 입력 복구를 차단하지 않는다.
- 임시 효과 노드·로그 행을 무제한 누적하지 않는다.
- 즉시 모드와 단계별 모드에서 결과가 같다.
- 대표 장면과 동시 공격·다수 로그의 최악 장면을 분리한다.
- frame time·CPU·GPU·메모리·로딩을 baseline과 비교한다.

목표 Windows 사양과 성능 프로파일은 `NOT_RUN`이다.

## 16. 현재 상태

### USER_CONFIRMED

- 기존 전장·배경·캐릭터·상단 HUD.
- 행동 슬롯·카드 트레이·상세·로그·진행.
- 행동 배치·대상 지정·기본 동기 판정.
