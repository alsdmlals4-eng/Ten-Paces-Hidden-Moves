# 십보강호 전투 연출 기획서

> 책임: 현재 T0 전투의 시선 흐름·계획·즉시 판정 피드백과 향후 단계별 무협 연출 계약  
> 규칙 원본: `docs/02_COMBAT_RULES.md`  
> UI 원본: `docs/07_COMBAT_UI_SPEC.md`  
> 범위·검증: `docs/05_COMBAT_POC_SPEC.md`, `docs/08_TEST_CHECKLIST.md`

## 1. 상태 구분

- `CURRENT`: 실제 POC가 지금 표현하는 방식.
- `IMPLEMENTED_FOR_REVIEW`: 코드에는 있으나 최신 사용자 확인 대기.
- `PLANNED`: 후속 STEP에서 구현.
- `PRESENTATION_TARGET`: T0 플레이테스트 뒤 필요성이 확인되면 제작.

현재 구현과 목표 연출을 같은 문장으로 설명하지 않는다.

## 2. 연출 목표

십보강호의 전투 연출은 화려함보다 다음을 우선한다.

1. 지금 무엇을 선택해야 하는지 이해한다.
2. 플레이어 4번·상대 7번의 시작 거리 3을 즉시 읽는다.
3. 배치한 수와 실제 판정의 관계를 신뢰한다.
4. 거리·방향·자원·대응·실패 원인을 다음 계획에 사용한다.
5. 계산을 방해하지 않으면서 무협 공방의 손맛을 제공한다.
6. 자산·모션·음향이 없어도 같은 결과를 읽는다.
7. 같은 단계 동시 공격과 이후 행동 중단을 시각 순서 때문에 오해하지 않는다.

## 3. 현재 T0 전투 흐름

라운드의 행동 묶음은 언제나 `3수 → 3수 → 4수`이며, 총 10수 뒤 다음 라운드로 넘어간다.
진행 패널은 `행동 묶음 판정` 캡션 아래 `배치 완료`, `행동·대상·자원 확인 필요`, `판정 완료`를 현재 상태에 맞게 표시한다.

```text
플레이어 4번·상대 7번과 시작 거리 3 확인
→ 양측 상태 확인
→ 현재 묶음 행동 배치
→ 이동 목적지·공격 방향 지정
→ 예상 기력·내력 확인
→ 진행
→ 대응 → 속공 → 이동 → 일반 공격을 엔진이 한 번 판정
→ 확정 HUD·위치·슬롯·문자열 로그 즉시 반영
→ 다음 묶음 또는 다음 라운드
```

현재 특징:

- 엔진은 `[진행]`에서 한 묶음의 확정 결과를 한 번 계산한다.
- 화면은 확정 `presentation_events`와 `timing_results`를 따라 대응 뒤 각 수의 속공·이동·일반 행동을 순차 재생한다.
- 캐릭터 위치와 HUD는 확정 snapshot에만 따라 갱신되며, UI는 결과를 재계산하지 않는다.
- 세부 판정 순서는 전투 기록과 결과 문구에서 확인한다.

## 4. 상태 소유 경계

| 계층 | 책임 | 금지 |
|---|---|---|
| 전투 도메인 | 위치·비용·대응·피해·자원·라운드 | 모션 완료 여부로 판정 변경 |
| 판정 반환 | 확정 state·해결 행동·방어 profile·로그 | 장면용 임의 피해 추가 |
| UI | 입력·선택 가능 상태·예상 자원·확정 결과 표현 | 피해·회복·승패 재계산 |
| VFX·애니메이션·오디오 | 확정 결과 재생 | 보상·저장·기세 직접 지급 |

연출 콜백에서 체력·기력·내력·기세를 바꾸지 않는다.

## 5. 첫 시선

전투 진입 뒤 권장 시선 순서:

```text
중앙 10칸 전장
→ 플레이어 4번·상대 7번과 사이 빈 칸
→ 상단 체력·기력·내력·절초 기세
→ 라운드·행동 묶음·판정 순서
→ 하단 현재 행동 슬롯
→ 기초 행동 8종
```

- 전장과 캐릭터가 카드 상세·로그보다 먼저 읽혀야 한다.
- 시작 거리 3은 별도 숫자 팝업이 없어도 타일 번호와 위치로 읽을 수 있어야 한다.
- 현재 편집 수와 미래 잠금 수를 즉시 구분한다.
- 상세와 기록은 필요할 때만 펼치는 오버레이다.
- 수묵 배경이 전장 칸·캐릭터·HUD 대비를 방해하지 않아야 한다.

## 6. 계획 단계 표현

### 6.1 카드 선택

CURRENT:

- 호버하면 상세 임시 보기.
- 클릭하면 상세 고정과 행동 선택.
- 같은 카드 재클릭 또는 빈 전장 클릭으로 해제.
- 태세·대응 조합 대기 상태.

PRESENTATION_TARGET:

- 상세 고정과 행동 선택을 별도 아이콘·테두리로 더 명확히 구분.
- 키보드 포커스와 선택 상태를 동일하게 표현.

### 6.2 슬롯 배치

- 빈 수에 행동 이름을 표시한다.
- 2슬롯 행동은 두 칸이 하나의 준비 행동임을 연결한다.
- 마지막 점유 수가 실제 실행 수임을 상세 또는 표식으로 전달한다.
- 묶음 경계 초과·점유 충돌은 배치하지 않고 원인을 로그에 기록한다.
- 점유 칸을 누르면 행동 전체를 해제한다.

### 6.3 대상 지정

시작 위치 기준 이동:

```text
이동 배치
→ 4번에서 3번·5번 녹색 표시
→ 도착 칸 선택
→ 슬롯에 방향·칸 번호
```

시작 위치 기준 보법:

```text
보법 배치
→ 4번에서 2번·3번·5번·6번 후보
→ 전장 밖·상대 점유 칸 제외
→ 선택 목적지 표시
```

공격:

```text
공격 배치
→ 붉은 좌·우 방향 후보
→ 방향 선택
→ 슬롯에 공격 방향
```

- 색뿐 아니라 테두리·문구·방향 표식을 함께 사용한다.
- 공격 후보는 실제 피격 칸이 아니라 방향 선택임을 오해하지 않게 한다.
- 시작 거리 3에서는 속공·강공이 당장 적중하지 않는다는 점을 결과 로그가 설명한다.
- 지정이 끝나면 다음 미지정 행동으로 이어질 수 있다.

### 6.4 자원 예상

IMPLEMENTED_FOR_REVIEW:

- 배치 즉시 기력·내력 예상값 갱신.
- 실행 순서에 따른 명상 회복.
- 부족 슬롯 문구와 진행 잠금.
- 행동 해제 시 재계산.

수용 기준:

- 예상값을 실제 확정값과 혼동하지 않는다.
- 판정 뒤 실제 값이 직전 예상과 일치한다.
- 부족 원인이 어느 행동인지 확인한다.

## 7. 진행 버튼·현재 판정 전환

CURRENT 상태 문구:

| 상태 | 표시 |
|---|---|
| 조건 미충족 | `행동·대상·자원 확인 필요` |
| 조건 충족 | `배치 완료` |
| 수별 결과 연출과 다음 묶음 반영 후 | `판정 완료` |

현재 구현은 다음 상태를 사용한다.

```text
committed
→ resolving
→ presenting_result
→ next_bundle_ready
```

```text
진행 클릭
→ 엔진 1회 판정
→ 응답 준비 및 각 수 snapshot 순차 재생
→ 다음 묶음 상태와 판정 완료 문구
```

- `timing_results`와 `presentation_events`는 표현 전용 입력이며 UI·VFX·오디오가 판정을 수정하지 않는다.

- `resolving` 동안 슬롯·카드·대상 입력을 명시적으로 잠근다.
- 스킵·빠른 재생을 제공해 반복 전투의 대기 시간을 줄인다.
- 애니메이션을 끄면 같은 상태 전환을 즉시 완료한다.

## 8. 판정 표현 순서

규칙과 같은 순서를 사용한다.

```text
대응 준비
→ 속공
→ 이동
→ 일반 공격·일반 행동
→ 피해·자원·기세 결과
```

### 8.1 Historical state (superseded on 2026-07-23)

### 8.1A Current implementation: authoritative timing playback

- `CombatResolutionEngine` resolves a committed bundle only once, then returns an immutable playback snapshot for response setup and for every actual timing in that bundle.
- The board applies those snapshots in sequence: response setup → 1수 → 2수 → 3수 (or the corresponding 4수~10수 range). Character tiles, HUD resources, damage, and interruption state therefore change when their timing is reached, not only after the full bundle.
- The UI consumes `timing_results.events` and never recomputes combat. Empty planning inputs remain locked through `committed`, `resolving`, and `presenting_result` until the next bundle is ready.
- Ultimate event bands use the approved RGBA ink-and-gold atlas. An ultimate remains visible for 0.70 seconds in normal playback, has a short scale pulse, and is still supported by fast playback, skip, and reduced-motion modes.
- Automated coverage verifies timing-1 movement state, timing-2 resource state, and actual ultimate VFX visibility during the authoritative playback path.

### 8.1B Previous implementation notes

- 단계별 애니메이션 대신 최종 위치·HUD와 문자열 로그를 즉시 표시한다.
- 같은 단계 공격은 판정 엔진에서 동시 처리된다.
- UI 표시 순서가 판정 순서를 바꾸지 않는다.

### 8.2 4/7 첫 묶음 기준 연출

통합 fixture 예:

```text
초기: 플레이어 4 / 상대 7
1수 이동 결과: 플레이어 5 / 상대 6
2수: 명상과 막기
3수: 인접 상태에서 양측 속공
```

- 이동 전과 이동 후 위치를 로그에서 구분한다.
- 플레이어와 상대가 같은 수에 이동하면 양측 결과를 같은 단계로 묶는다.
- 3수 공격이 적중하는 이유는 `이동 후 인접`으로 설명한다.

### 8.3 PRESENTATION_TARGET

대응:

- 막기: 자세·푸른 표식·감소량.
- 회피: 잔상 또는 명확한 회피 표식.
- 태세+막기: 묶음 지속 강화 방어 표식.
- 태세+회피: 묶음 지속 회피 표식.

속공:

- 이동 전에 공격 방향과 실행 수를 강조.
- 적중·방향 실패·사거리 실패를 구분.

이동:

- 출발·도착 칸을 먼저 강조.
- 실패 이동을 실행했다가 되돌리는 연출 금지.
- 동일 목적지 충돌은 양측 표식과 원인 문구 제공.

일반 공격:

- 이동 후 실제 거리와 방향을 표현.
- 같은 단계 쌍방 공격을 한쪽 선공으로 취소된 것처럼 연출하지 않음.
- 양쪽 피해를 동시에 또는 같은 판정 묶음으로 표시.

일반 행동:

- 명상 회복과 단독 태세 저장을 공격과 구분.

## 9. 한 수의 결과 피드백

한 수는 다음 질문에 답해야 한다.

- 누가 어떤 행동을 했는가?
- 어느 판정 단계인가?
- 출발·도착 위치는 어디인가?
- 방향과 거리가 유효한가?
- 회피·막기·태세 중 무엇이 적용됐는가?
- 원피해·감소량·최종 피해는 얼마인가?
- 위치·기력·내력·기세가 어떻게 변했는가?
- STEP 11 이후 어떤 행동이 왜 중단됐는가?

권장 로그:

```text
[1라운드 1묶음 판정]
[1수] 강호낭인이 4번에서 5번으로 이동했다.
상대는 7번에서 6번으로 이동했다.
[3수] 강호낭인이 오른쪽으로 속공했다.
거리 1에서 적중했다.
```

피할 표현:

```text
priority=20 / emitted=true / damage_result=2
```

개발 진단은 별도 로그나 디버그 snapshot에서 확인한다.

## 10. 전투 기록 패널

CURRENT:

- 기본 접힘.
- `기록` 버튼으로 펼침·접음.
- 문자열 로그 추가·초기화.
- 사용자가 직접 스크롤.

현재 없는 기능:

- 새 로그 후 자동 최하단 이동.
- 사용자가 과거 로그를 읽을 때 자동 이동 일시 정지.
- 로그 필터·검색·수별 접기.
- 구조화 event ID와 현지화.

PRESENTATION_TARGET:

- 라운드→묶음→수 계층.
- 계획 로그와 판정 로그 구분.
- 최신 로그 자동 이동과 사용자 스크롤 보호.
- 한 수 요약과 상세 수치 펼치기.

향후 읽기 성공 표식 후보:

- 간파: 정확한 대응.
- 파훼: 준비 행동 중단.
- 선점: 이동 후 위치 예측 공격.
- 받아냄: 큰 피해를 막기로 감소.
- 허초: 상대 대응을 낭비시킴.

표식은 현재 구현이 아니며 STEP 14에서 필요성을 검증한다.

## 14. 접근성 폴백

### 현재

## 수묵 석양 UI와 단순 모션 구현 기록 (2026-07-23)

- 전장은 전용 수묵 석양 PNG를 사용하고, 저대비 배경 위에 10칸 타일·전투원·금빛 절초 강조를 분리한다. 카드·타일·아이콘의 SVG 정보 계층은 래스터로 대체하지 않는다.
- 좌우 상태 패널은 강호낭인/무명 검객의 프로젝트 전용 초상을 표시하며, 체력·기력·내력·기세는 기존 판정 상태만 표현한다.
- 절초 목록은 플레이어 기세 게이지 하단에 놓여, 기세 5/5·연속 빈 수·현재 입력 잠금 여부를 버튼별 텍스트와 비활성 상태로 설명한다.
- HUD 하위 패널의 최소 크기는 960×640 전장에서도 한 화면에 남도록 축소 가능하며, 960×640/1440×900 headless 배치 회귀를 둔다.
- `presentation_events`와 `timing_results`는 한 번 계산된 엔진 결과다. UI는 대응 뒤 1수부터 순차적으로 상태를 적용하고, 이동은 짧은 발 앵커 이동, 공격/절초는 짧은 전진 공격, 대기는 가벼운 호흡 움직임으로 표현한다.
- 이 모션은 판정·피해·위치·자원을 다시 계산하거나 변경하지 않는다. `모션 감소`는 이동/공격 트윈을 생략하고 확정 결과와 텍스트/로그/VFX 접근 경로는 유지한다.
- `즉시 완료`는 진행 중인 이벤트의 일반 타이머를 다음 프레임에 탈출시킨다. 따라서 절초의 최대 0.70초 VFX 대기를 끝까지 기다리지 않으며, 현재 결과 텍스트와 VFX를 숨기되 확정 상태·로그·자원·기세는 바꾸지 않는다.
- 전장 전투원은 프로젝트 전용 전신 수묵 원화를 사용한다. 각 PNG는 크로마키를 RGBA로 변환해 모서리 알파·투명 픽셀 수를 검수했고, 원화의 발을 기존 타일 앵커에 맞춰 대기·이동·공격 transform을 적용한다.
- 표준 버튼과 슬라이더는 흰색 2px 포커스 링과 옅은 면 채움으로 Tab 위치를 표시한다. 카드·수 슬롯·대상 타일의 기존 자체 포커스 테두리와 함께 색각에만 의존하지 않는 입력 경로를 만든다.
- Tab 순서는 카드 8종 → 수 슬롯 1~10 → 대상 타일 1~10 → 진행 → 빠른 재생·즉시 완료·모션 감소·소리·음량 → 카드로 명시적 순환한다. Windows UI Automation은 이 제어들의 한국어 이름·역할 노출을 확인했고, 실제 보조기기 사용자의 전체 조작성은 별도 수동 Gate로 남긴다.
- 포커스 가능한 전투 조작 요소에는 Godot Control의 한국어 `accessibility_name`·`accessibility_description`을 함께 설정한다. 이는 읽기 도구에 역할과 조작 결과를 제공하는 구현 증거이며, 운영체제 보조기기와의 실제 음성 출력 검수는 대체하지 않는다.
- `verify_combat_pointer_lock.gd`는 실제 `InputEventMouseButton`을 카드·수 슬롯·절초 버튼에 전달한다. `resolving`에서는 기존 선택을 바꾸지 않고, 수 예약과 절초 기세 소비도 발생하지 않아야 한다.
- 자동 검증과 HiGodot Windows 런타임의 4/7 시작 렌더, 절초 목록 활성·예약·대상 입력, 대시 후 5·6번 위치, 수묵·금빛 VFX와 `사거리 실패` 결과 표시, `소리:끔` → `소리:켬` 토글 텍스트는 완료했다. 진행 버튼 직후 `resolving`·`locked=true`와 카드 선택 핸들러 차단도 런타임에서 확인했다. Headless는 파공검기 처치 뒤 `combat_ended` 입력 잠금·전투 불능 문구·패배음 요청, 확정 기세 증가·금속 충돌/완전 막기 SFX 요청과 밀착·파공검기 VFX·로그 52개 장면의 120프레임 성능 표본도 확인했다. 대표 시작 장면의 5개 표본은 145 FPS·267 draw call·약 54.1MB video memory이고, RTX 3050 최악 장면은 평균 17.11ms·378 draw call·69.6MB video memory다. Windows AudioStreamWAV 재생·음소거 즉시 정지·PCM 음량과 UI Automation 노출도 확인했다. 주관적 청취/모션과 실제 보조기기 사용성, Release 사양은 `NOT_RUN`이다.
## Issue #11 연출 구현 갱신 (2026-07-23)

화면 상태는 `planning → committed → resolving → presenting_result → next_bundle_ready`로 기록되며, 체력 0이면 마지막 결과 뒤 `combat_ended`로 전환해 추가 계획 입력을 잠근다. 절초 버튼은 기세 부족·연속 빈 수 부족·판정 중을 tooltip으로 별도 설명한다. `planning`의 절초 점유 수 슬롯은 클릭 또는 Enter로 전체 예약을 취소하고 기세 5를 반환한다는 tooltip을 제공하며, `committed` 이후에는 입력 잠금과 함께 취소·환불도 막는다. 결과 라벨은 60px 높이·한국어 단어 단위 줄바꿈으로 확정 이벤트의 중단·방향/사거리 실패·피해·전투 불능을 표시하고, 빠른 재생·즉시 완료·모션 감소·음소거·음량을 제공한다. 절차적 SFX는 기세 충전·절초 예약·검풍·금속 충돌·강타·완전 막기·회피·중단·전투 불능 역할음을 확정 snapshot과 이벤트에서 요청한다. `assets/vfx/ultimate_ink_gold_sprite_sheet_rgba.png`는 RGBA 알파 검수 뒤 활성화됐으며, 세 절초가 각 밴드를 AtlasTexture로 재생한다.

최신 Windows 검증에서는 절차적 SFX가 실제 `AudioStreamWAV`를 만들고 재생하며, 음소거는 현재 재생을 즉시 정지하고 음량은 PCM 진폭에 반영됨을 확인했다. 프로젝트는 `accessibility/general/accessibility_support=1`로 Godot 접근성 갱신을 항상 활성화하고, Windows UI Automation 트리에 전투 카드·수 슬롯·진행·절초·재생/음향 제어가 한국어 이름과 역할로 노출됨을 확인했다.
