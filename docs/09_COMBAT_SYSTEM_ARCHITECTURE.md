# 전투 시스템 아키텍처

> 원칙: 무공이 전투 규칙을 새로 만들지 않는다. 공용 행동과 무기 규칙을 먼저 해상하고, 유파 무공·대성은 정해진 수정 지점만 바꾼다.

## 1. 기술·출시 기준

- 엔진은 **Godot 4.7.1 + GDScript**를 사용한다.
- 출시 목표는 Steam의 Windows 데스크톱 빌드다.
- Steamworks 상점·업적·클라우드 연동은 현재 세로 슬라이스 범위에서 제외한다.
- 전투 규칙은 Godot 씬과 분리된 순수 GDScript 시뮬레이터로 실행한다.
- UI는 입력을 수집하고 시뮬레이터가 반환한 구조화 이벤트를 재생하며, 전투 결과를 직접 계산하지 않는다.

## 2. 다섯 층

| 층 | 책임 | 예시 |
|---|---|---|
| 공용 행동 | 전투에서 가능한 기본 선택과 슬롯 해상 | 접근, 후퇴, 공격, 막기, 회피, 밀치기, 무기 교체, 재장전 |
| 무기 규칙 | 사거리, 행동력·기력 비용, 막기·밀치기 효율, 재장전 | 권법 0, 창 2, 활 3~5·재장전 |
| 보조장비 | 기존 행동에 붙는 제한 횟수의 무행동 효과 | 암기벨트, 화살통, 방패 |
| 유파 무공 | 공용 행동의 조건·효율·결과 수정 | 금강호신이 막기를 강화, 독침이 공격에 독 부여 |
| 대성 | 문파 규칙의 단계 해금과 외문 수련 | 1~5성, 외문 규칙 슬롯 |

## 3. 시뮬레이션 계층

| 구성 요소 | 책임 |
|---|---|
| `CombatState` | 위치, 체력, 행동력, 기력, 내공, 장전, 상태, 장비, 대성 상태 보관 |
| `ActionOrder` | 6슬롯 행동과 장비 부착 행동을 검증 가능한 형태로 보관 |
| `CombatResolver` | 슬롯 하나를 해상하고 전투 이벤트와 다음 상태 반환 |
| `RoundResolver` | 6개 슬롯과 라운드 종료 회복·독·마화를 순서대로 실행 |
| `FactionModifier` | 유파·대성 효과가 수정할 수 있는 수치와 조건 제공 |
| `AiPlanner` | AI의 공개 정보와 성향만 받아 잠긴 행동열 생성 |

전투 이벤트는 UI가 결과를 추론하지 않도록 `move`, `guard`, `evade`, `hit`, `push`, `status`, `reload`, `resource`, `invalid_action`의 구조화된 기록으로 반환한다.

## 4. 데이터 계층

문파·무기·무공·보조장비·AI 성향·대진은 Godot Resource 데이터로 저장한다. 시뮬레이터는 표시 이름이 아니라 데이터 ID와 허용된 수정자만 참조한다.

| 데이터 | 최소 필드 |
|---|---|
| `WeaponData` | ID, 사거리, 피해, 행동력·기력 비용, 막기·밀치기 효율, 재장전 규칙 |
| `FactionData` | ID, 분류, 기본 무기, 시작 무공, AI 성향, 대표 거리 |
| `MartialArtData` | 문파 ID, 1~5성 효과, 허용 수정자, 해금 수련도 |
| `AccessoryData` | 호환 무기, 사용 횟수, 부착 대상 행동, 비용 면제 효과 |
| `OpponentProfile` | 세력, 난이도 단계, 보유 무공·장비, AI 성향 |
| `TournamentStageData` | 경기 번호, 단계, 상대 풀, 보상 수, 정탐 가능 여부 |

## 5. 엔진 계약

1. 모든 행동은 공용 행동 타입으로 기록한다.
2. 무기는 행동 가능 여부와 기본 수치를 판정한다.
3. 장비·유파·대성은 허용된 수정값만 제공한다: 비용, 사거리, 피해, 경감, 이동량, 상태 부여, 재장전 상태, 사용 횟수.
4. 문파 전용 행동 타입·전용 슬롯·전용 해상 순서는 만들지 않는다.
5. 상태는 독·마화·장전·방패 등 공통 상태만 쓴다.
6. 모든 수정값은 결과 로그에서 원인과 함께 표시한다.
7. AI는 플레이어의 비공개 행동열을 읽지 않고, 행동열 생성 뒤 수정하지 않는다.
8. 유효하지 않은 행동은 상태를 손상시키지 않고 `invalid_action` 이벤트를 남긴 뒤 대기로 처리한다.

## 6. 슬롯 해상 순서

1. 양측의 해당 슬롯 행동과 보조장비를 확인한다.
2. 행동 가능 여부와 비용을 판정한다.
3. 이동을 동시에 처리한다.
4. 막기·회피·방패 효과를 적용한다.
5. 이동 뒤 최종 거리를 기준으로 공격·밀치기를 판정한다.
6. 피해·독·마화·장전 등 상태를 동시에 반영한다.
7. 라운드 종료 시 기력·내공 회복과 독·마화 피해를 처리한다.

## 8. 12세력 적용

| 계열 | 공용 행동의 주 수정 지점 |
|---|---|
| 근접 압박 | 소림 권법, 오독 독장: 거리 0 공격·상태 |
| 검·도 기동 | 화산·무당·마교: 접근·후퇴·회피·체력 대가 |
| 거리 2 장악 | 개방 곤법, 양가장 창법, 관군 창방패: 밀치기·막기·회수 |
| 원거리 견제 | 당문 암기, 백우궁 활: 재장전·보조장비·거리 유지 |
| 자원전 | 아미 점혈·의술, 녹림 약탈도, 무당 심법, 마교 마화: 기력·내공·체력 변환 |

## 9. 저장과 검증

- 저장 데이터는 포맷 버전, 회차 상태, 선택 문파, 현재 경기, 상대 풀, 보유 무공·수련도·장비·명망·재도전 사용 여부를 보관한다.
- 전투 시뮬레이터는 Godot headless 테스트로 거리·동시 이동·피해·독·마화·재장전·대성·AI 잠금을 검증한다.
- 12세력과 9전 회차는 통합 테스트에서 결승까지 진행되어야 한다.
- 시작 6세력과 추가 6세력은 모두 같은 엔진 계약으로 동작한다.
- 새로운 문파는 공용 엔진 수정 없이 데이터와 허용된 수정자로 추가할 수 있어야 한다.
