# 십보강호 전투 UI·아트 명세

> 책임: 현재 T0 전투 화면·입력·카드·상태·결과·접근성 표현 계약  
> 판정 원본: `docs/02_COMBAT_RULES.md`  
> 범위: `docs/05_COMBAT_POC_SPEC.md`  
> 연출: `docs/10_COMBAT_PRESENTATION_PLAN.md`

## 1. UI 목표

플레이어가 한 화면에서 다음 질문에 답할 수 있어야 한다.

1. 양측은 어디에 있고 거리·밀착 상태는 무엇인가?
2. 체력·기력·내력·기세·상태는 얼마인가?
3. 지금 어느 라운드·묶음·수인가?
4. 어떤 행동이 준비·실행·중단될 예정인가?
5. 이동 목적지·공격 방향·절초 예약은 확정됐는가?
6. 진행할 수 없다면 무엇이 부족한가?
7. 합·방어·회피·필중·중단 결과가 왜 발생했는가?
8. 종료 뒤 무엇을 바꿔 다시 시도할 수 있는가?

전투 중 실패 원인과 판정 결과는 직접적이고 반복 가능한 용어를 사용한다.

## 2. 상태 등급

- `CURRENT`: PR #7 Godot UI에 존재한다.
- `TECHNICALLY_VERIFIED`: headless·Windows·UI Automation 등 기술 증거가 있다.
- `HUMAN_NOT_RUN`: 실제 사용자 이해·보조기기 사용성·주관적 읽기성 검증 전이다.
- `T1_LATER`: 현재 T0 화면에 넣지 않는다.

## 3. 화면 구조

| 영역 | 현재 표시 |
|---|---|
| 상단 좌측 | 플레이어 초상·이름·이명·체력·기력·내력·상태 |
| 상단 중앙 좌측 | 플레이어 기세 0~5와 절초 목록 3종 |
| 상단 중앙 | 라운드·묶음·현재 수·판정 순서 |
| 상단 중앙 우측 | 상대 기세 0~5 |
| 상단 우측 | 상대 초상·이름·이명·체력·기력·내력·상태 |
| 중앙 | 1~10번 전장과 전신 수묵 전투원 |
| 하단 상단 | `3수 → 3수 → 4수` 슬롯과 진행·재생 제어 |
| 하단 하단 | 기초 행동 8종 |
| 좌측 오버레이 | 카드 상세 |
| 우측 오버레이 | 접이식 전투 기록 |

- 전장과 캐릭터가 첫 시선이다.
- 시작 4/7과 거리 3을 타일 번호·빈 공간·발 앵커로 읽는다.
- 같은 칸에서는 `[밀착]` 텍스트·상태·로그로 거리 0을 표시한다.
- 하단 카드는 손패가 아니라 현재 사용 가능한 행동 목록이다.
- 덱·드로·버린 카드·행동력을 표시하지 않는다.

## 4. 시각 자산

- 배경: `assets/backgrounds/twilight_ink_duel_v1.png`.
- 전투원: 플레이어·상대 RGBA 전신 수묵 자산.
- 초상: 플레이어·상대 프로젝트 전용 수묵 초상.
- 카드: 기초 카드 atlas·badge·비용 아이콘.
- 절초: 수묵·금빛 RGBA 이벤트 atlas.
- 자산 출처·역할·라이선스·폴백은 `assets/ASSET_MANIFEST.json`이 소유한다.

자산이 로드되지 않아도 기본 도형·텍스트·상태 이름으로 기능 정보를 유지한다.

## 5. 전장 칸 상태

색뿐 아니라 형태·테두리·문구·접근성 이름을 함께 사용한다.

| 상태 | 의미 | 입력 |
|---|---|---|
| 기본 | 현재 선택과 무관 | 포인터를 가로채지 않음 |
| 플레이어/상대 | 현재 점유 | 상태 읽기 |
| 밀착 | 양측 공동 점유 | 거리 0 설명 |
| 이동 가능 | 유효 목적지 | 클릭·Enter |
| 공격 방향 | 좌·우 방향 후보 | 클릭·Enter |
| 선택 | 확정 목적지·방향 | 슬롯 문구와 동시 표시 |
| 비활성 | 전장 밖·교환·통과·범위 밖 | 선택 불가와 이유 |

- 정지 상대 칸과 공동 빈 목적지는 현행 규칙에 따라 이동 후보가 될 수 있다.
- 자리 교환·상대 통과는 후보에서 제외한다.
- 공격 후보 칸은 실제 피격 칸이 아니라 방향 선택 인터페이스다.
- `[밀착]`에서는 방향과 무관한 자동 유효 사거리라는 결과를 설명한다.

## 6. 상단 HUD

고정 정보 순서:

```text
플레이어 상태 | 플레이어 기세·절초 | 라운드·묶음 | 상대 기세 | 상대 상태
```

### 6.1 상태 패널

- 체력·기력·내력을 `현재/최대`로 표시한다.
- 공격력은 절초 계산·상세에서 필요한 경우 표시한다.
- `[밀착]`, `[강건]`, 준비·중단·전투 불능 등 상태를 텍스트 칩과 설명으로 표시한다.
- 계획 예상 기력·내력은 확정값과 다른 표현을 사용한다.
- 종료 뒤 최종 상태를 유지하고 계획 입력을 잠근다.

### 6.2 절초 기세와 목록

- 기세는 정확히 5칸이다.
- 양측은 0/5에서 시작한다.
- 플레이어 절초 목록은 planning 중 기세 게이지 바로 아래에 항상 표시한다.
- 십보 유파·단악결·파공검기의 슬롯·사거리·피해와 비활성 이유를 표시한다.
- 기세가 정확히 5이고 연속 빈 슬롯이 있을 때만 선택 가능하다.
- 예약 즉시 기세가 0이 되고 슬롯에 `[절초 예약]`을 표시한다.
- `[진행]` 전 취소 시 기세 5 복원을 즉시 표시한다.
- `committed` 이후 실패·중단에는 환불하지 않는다.

### 6.3 라운드·묶음

```text
제 1 라운드
행동 묶음 1/3
1수 · 2수 · 3수 선택
대응 → 속공 → 이동 → 일반 공격
```

하단 진행 영역은 중복된 총 진행 문구를 최소화한다.

## 7. 행동 슬롯

### 7.1 구조

```text
1묶음: 1수 · 2수 · 3수
2묶음: 4수 · 5수 · 6수
3묶음: 7수 · 8수 · 9수 · 10수
```

### 7.2 상태

- 빈 슬롯.
- 현재 편집 가능.
- 미래 묶음 잠금.
- 일반 행동 점유.
- `[준비]`.
- `[실행]`.
- 대상 지정 대기.
- 자원 부족.
- `[절초 예약]`.
- 판정 완료.
- 피해 중단.
- 전투 불능 취소.

### 7.3 표시 내용

- 카드·절초 이름.
- 점유 연결선과 실행 수.
- 이동 방향·도착 칸.
- 공격 방향.
- 결합 행동 `태세+막기`, `태세+회피`.
- 예약·중단·실패 원인.

다중 슬롯의 앞 수는 준비, 마지막 수는 실행임을 문구·형태로 구분한다.

## 8. 행동 선택과 대상 지정

현재 입력 흐름:

```text
카드 또는 절초 선택
→ 현재 묶음의 빈 슬롯 선택
→ 필요한 경우 전장 목적지·방향 선택
→ 슬롯·예상 자원·예약 상태 확인
→ 진행
```

- 카드 클릭은 상세 고정과 행동 선택을 제공한다.
- 같은 카드 재클릭 또는 빈 전장 클릭으로 해제할 수 있다.
- 대상 지정 중에는 입력 순서를 보존한다.
- 이동은 도착 칸을 선택한다.
- 공격은 좌·우 방향을 선택한다.
- 절초는 필요한 연속 슬롯과 방향을 선택한다.
- 배치 해제는 예상 자원과 기세 예약을 즉시 다시 계산한다.

## 9. 진행·재생·재시작

### 9.1 진행

| 조건 | 상태 문구 |
|---|---|
| 슬롯·대상·자원 미완성 | `행동·대상·자원 확인 필요` |
| 현재 묶음 완료 | `배치 완료` |
| 결과 재생 중 | `판정 중` |
| 다음 묶음 준비 | `판정 완료` |
| 전투 종료 | 승리·패배·무승부 결과 |

엔진은 진행 요청마다 묶음을 한 번 계산한다. 화면은 `committed → resolving → presenting_result → next_bundle_ready` 동안 입력을 잠근다.

### 9.2 재생 제어

- 빠른 재생·즉시 완료·모션 감소는 같은 확정 결과를 사용한다.
- 즉시 완료는 남은 연출 대기를 끝내지만 상태·로그·자원·기세를 바꾸지 않는다.
- 모션 감소는 이동·공격 트윈을 생략해도 텍스트·로그·상태 변화를 유지한다.

### 9.3 재시작

`결전 다시 시작`은 종료 때만 표시한다.

- 4/7·30 체력·5 기력·4 내력·0/5 기세로 복원한다.
- 라운드·묶음·슬롯·대상·태세·강건·예약·로그·연출·오디오 상태를 초기화한다.
- 숨겨진 버튼은 Tab 순서를 가로채지 않는다.

## 10. 합·대응·중단 결과 UI

### 10.1 합

- 양측 행동·실행 수·원공격력을 같은 결과 묶음에 표시한다.
- 합 승리·차이 피해·동점 상쇄를 구분한다.
- 방어·회피 적용 전후 값을 분리한다.

### 10.2 방어·회피·필중

- 막기: 원피해→방어도 차감→같은 수 반감→최종 피해.
- 태세+막기: 묶음 방어도 6.
- 회피: 같은 수 또는 묶음 전체 회피 범위.
- 필중: 회피만 무시하며 다른 조건은 유지한다는 이유.

### 10.3 중단·강건

- 준비·실행·중단 상태를 슬롯과 결과 이벤트에서 일치시킨다.
- 같은 수 미실행 행동만 중단됐다는 이유를 표시한다.
- `[강건]`이 중단을 막아도 피해는 받았음을 분리해 표시한다.
- 전투 불능으로 이후 수가 취소된 경우를 일반 중단과 구분한다.

UI는 결과를 계산하지 않고 `timing_results`와 `presentation_events`를 소비한다.

## 11. 카드·상세 정보 계층

카드 본체:

- 소속 `[기초]` 또는 절초.
- 이름·기능·사거리.
- 슬롯·기력·내력 또는 기세 조건.
- 대표 원화.

상세 패널:

1. 이름·소속·기능·사거리.
2. 대상·방향·이동.
3. 피해·방어·회복·계수.
4. 사용 조건.
5. 효과 설명.
6. 태그.
7. 비용과 슬롯.
8. 플레이버 문구.

같은 전문을 카드·호버·고정 상세에 반복하지 않는다.

## 12. 전투 기록

기본 접힘 상태이며 수동으로 펼친다.

표시 대상:

- 배치·해제·배치 불가.
- 대상·방향·절초 예약·취소.
- 자원 부족.
- 이동·밀착·공동 목적지·교환/통과 실패.
- 합·동점·방어·회피·필중.
- 피해·자원·기세·상태 변화.
- 준비·실행·중단·전투 불능.
- 승패·무승부·재시작.

개발 변수보다 플레이어가 다음 계획에 사용할 원인 문장을 우선한다.

## 13. 입력·접근성

### 현재 기술 계약

- 카드 8종 → 절초 3종 → 수 슬롯 1~10 → 대상 타일 1~10 → 진행 → 재생 제어 → 재시작(표시 시) → 모션 감소 → 소리 → 음량 → 카드 순으로 명시적 포커스를 구성한다.
- `Enter`/`ui_accept`는 카드 선택·슬롯 배치·대상 확정·예약 취소에 포인터와 같은 동작을 한다.
- 카드·슬롯·타일·진행·재생·음향 제어에 한국어 `accessibility_name`과 설명을 제공한다.
- 포커스는 흰 테두리·면 변화로 항상 보이게 한다.
- 색 외에 텍스트·아이콘·형태를 제공한다.
- 긴 한국어 결과 문구와 960×640·1440×900 배치를 회귀 검사한다.
- 모션 감소와 음향 끄기에서도 같은 정보가 남는다.

### 사람 검증 상태

Windows UI Automation 노출은 기술적으로 확인했지만 다음은 `HUMAN_NOT_RUN`이다.

- 실제 보조기기 사용자의 전체 흐름.
- 음성 출력의 이해 가능성.
- 키보드만으로 한 판 완료.
- 주관적 모션·음향 읽기성.
- 색각·저시력 사용자의 실제 장벽.

법적 접근성 준수 인증으로 표현하지 않는다.

## 14. 현재 검증 상태

```yaml
implementation: CURRENT
windows_render_and_input: TECHNICALLY_VERIFIED
ui_automation_metadata: TECHNICALLY_VERIFIED
minimum_layout_regression: TECHNICALLY_VERIFIED
human_comprehension: NOT_RUN
assistive_technology_usability: NOT_RUN
subjective_motion_audio_readability: NOT_RUN
release_performance: NOT_RUN
```

## 15. T1 이후 화면

T0 사람 검증과 코어 승인 뒤 별도 범위로 구현한다.

- 상대 이명·풍문·전적·정탐.
- 제약 선택.
- 전투 후 보상·수련.
- 무공·심법·성급·진의.
- 대진 진행과 5전 데모 결말.

미래 콘텐츠 정보를 현재 T0 화면 완료 조건으로 선반영하지 않는다.
