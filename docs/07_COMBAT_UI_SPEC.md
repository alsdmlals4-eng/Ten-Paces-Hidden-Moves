# 십보강호 전투 UI·아트 명세

> 책임: 현재 T0 전투 화면·카드·행동 계획·상태·접근성 표현 계약  
> 판정 원본: `docs/02_COMBAT_RULES.md`  
> POC 범위: `docs/05_COMBAT_POC_SPEC.md`  
> 연출 흐름: `docs/10_COMBAT_PRESENTATION_PLAN.md`

## 1. UI 목표

플레이어가 한 화면에서 다음 질문에 답할 수 있어야 한다.

1. 양측은 어디에 있고 현재 거리는 얼마인가?
2. 체력·기력·내력·절초 기세는 얼마인가?
3. 지금 어느 라운드·묶음·수인가?
4. 어떤 행동을 어느 수에 배치했는가?
5. 이동 목적지와 공격 방향은 지정됐는가?
6. 진행할 수 없다면 무엇이 부족한가?
7. 판정 결과가 왜 발생했는가?

전투 중에는 무협식 은유보다 판정 명료성을 우선한다.

## 2. 전투 화면 구조

| 영역 | 표시 내용 |
|---|---|
| 상단 좌측 | 플레이어 초상·체력·기력·내력·상태 |
| 상단 중앙 좌측 | 플레이어 절초 기세, 최대 5칸 |
| 상단 중앙 | `제N라운드`, 현재 묶음과 수 |
| 상단 중앙 우측 | 상대 절초 기세, 최대 5칸 |
| 상단 우측 | 상대 초상·체력·기력·내력·공개 상태 |
| 중앙 전장 | 1~10번 칸, 플레이어 3번·상대 8번 시작, 선택 가능 칸 |
| 하단 상단 | `3수 → 3수 → 4수` 행동 슬롯과 `[진행]` |
| 하단 하단 | 이동·보법·막기·회피·속공·강공·명상·태세 8종 |
| 좌측 오버레이 | 카드 상세 패널 |
| 우측 오버레이 | 접이식 전투 기록 |

- 전장과 캐릭터가 화면의 중심이다.
- 좌우 세로 캐릭터 패널로 전장 너비를 침범하지 않는다.
- 덱·손패·드로·남은 카드·버린 카드 수를 표시하지 않는다.
- 하단 행동 카드는 항상 사용 가능한 수의 목록이며 무작위 손패가 아니다.

## 3. 전장 상태

각 칸은 색뿐 아니라 형태·아이콘·문구를 함께 사용한다.

| 상태 | 의미 |
|---|---|
| 기본 | 현재 선택과 무관한 칸 |
| 플레이어 | 플레이어 점유 |
| 상대 | 상대 점유 |
| 이동 가능 | 현재 이동 행동의 유효 목적지 |
| 공격 방향 | 현재 공격 행동이 지정할 수 있는 좌·우 방향 |
| 선택 | 확정한 목적지 또는 방향 |
| 비활성 | 전장 밖·점유·사거리·규칙상 선택 불가 |

- 이동 가능 칸은 녹색 계열과 이동 표식을 사용한다.
- 공격 방향 칸은 적색 계열과 공격 표식을 사용한다.
- 기본 칸은 포인터 입력을 가로채지 않는다.
- 대상 선택 중 활성화된 칸만 클릭 입력을 받는다.

## 4. 상단 HUD

고정 순서:

```text
플레이어 상태 | 플레이어 절초 기세 | 라운드 | 상대 절초 기세 | 상대 상태
```

### 상태 패널

- 체력·기력·내력은 현재값/최대값으로 표시한다.
- 행동 계획 중에는 기력·내력의 예상값을 즉시 보여준다.
- 예상값은 실제 상태와 시각적으로 구분한다.
- 행동 배치 해제 시 예상값을 다시 계산한다.

### 절초 기세

- 최대 5칸이다.
- 플레이어와 상대 모두 같은 형태를 사용한다.
- 현재 POC fixture의 4/5·3/5는 게이지 표현 검증용이며 정식 시작 밸런스가 아니다.
- 획득·소모·절초 사용의 완성 규칙은 후속 단계에서 별도 검증한다.

### 라운드

- 상단 중앙은 `제1라운드`처럼 라운드를 표시한다.
- 하단 진행 영역은 `라운드 1`만 표시하며 `(5/10수)` 같은 중복 문구를 사용하지 않는다.
- 현재 수는 슬롯 상태로 확인한다.

## 5. 행동 슬롯

### 구조

```text
1묶음: 1수 · 2수 · 3수
2묶음: 4수 · 5수 · 6수
3묶음: 7수 · 8수 · 9수 · 10수
```

상태:

- 지난 수
- 현재 편집 가능
- 점유
- 대상 지정 대기
- 자원 부족
- 미래 묶음 잠금
- 판정 완료

### 배치

- 1슬롯 행동은 한 수를 점유한다.
- 2슬롯 행동은 현재 묶음의 연속된 두 수를 점유한다.
- 묶음 경계를 넘어갈 수 없다.
- 다중 슬롯의 어느 점유 칸을 클릭해도 행동 전체를 해제한다.
- 실제 실행 수는 마지막 점유 수다.

### 슬롯 표시

- 카드 이름
- 다중 슬롯 점유 연결
- 이동 목적지: `← 2번`, `→ 4번`
- 공격 방향: `← 공격`, `→ 공격`
- 미지정 대상
- 자원 부족
- 결합 행동: `태세+막기`, `태세+회피`

## 6. 진행 버튼

`[진행]`은 다음 조건을 모두 만족할 때만 활성화한다.

- 현재 묶음의 모든 슬롯 점유
- 이동·공격 대상 지정 완료
- 다중 슬롯 점유 유효
- 실행 순서 기준 예상 기력·내력 유효

버튼 상태:

- 비활성: `판정 전`
- 요청 직후: `진행 요청됨`
- 판정 중: 입력 잠금
- 다음 묶음: 다시 계획 가능

판정 시작 뒤 현재 묶음은 수정할 수 없다.

## 7. 카드 템플릿

### 고정 정보 계층

| 위치 | 정보 |
|---|---|
| 좌측 상단 | 소속 `[기초]` 또는 문파·세력 |
| 상단 중앙 | 사거리 |
| 우측 상단 | 기능 배지: 이동·공격·대응·회복·강화 |
| 제목부 | 카드 이름 |
| 중앙 | 카드 원화 |
| 하단 | 행동 슬롯·기력·내력 |

- `[속공]`, `[집중]`, `[강건]`, `[검법]`, `[보법]`은 상세 태그다.
- 행동력은 사용하지 않는다.
- 공통 `막기 경감` 필드를 카드 하단에 두지 않는다.
- 프레임·소속·사거리·기능 배지·비용·원화·문구·데이터를 분리한다.
- 이미지에 텍스트를 구워 넣지 않는다.

## 8. 기초 행동 8종 표시

| 행동 | 기능 | 대상 표시 |
|---|---|---|
| 이동 | 이동 | 좌·우 1칸 녹색 목적지 |
| 보법 | 이동 | 좌·우 1~2칸 녹색 목적지 |
| 막기 | 대응 | 자신, 대상 선택 없음 |
| 회피 | 대응 | 자신, 대상 선택 없음 |
| 속공 | 공격 | 좌·우 거리 1 붉은 방향 |
| 강공 | 공격 | 좌·우 거리 1~2 붉은 방향 |
| 명상 | 회복 | 자신, 대상 선택 없음 |
| 태세 | 강화 | 자신 또는 대응 결합 |

## 9. 카드 상세

### 임시 보기

- 마우스를 올리면 표시한다.
- 마우스를 벗어나면 닫는다.

### 고정 보기

- 카드를 클릭하면 고정한다.
- 같은 카드 재클릭 또는 빈 전장 클릭으로 닫는다.
- 행동 배치 선택과 상세 고정 상태를 구분한다.

표시 순서:

1. 이름·소속·기능·사거리
2. 대상
3. 피해·이동량·방어도·회복 등 핵심 수치
4. 사용 조건
5. 효과 설명
6. 태그
7. 행동 슬롯·기력·내력
8. 플레이버 문구

## 10. 전투 기록

기본적으로 접힌 `기록` 버튼으로 제공한다.

로그는 개발 이벤트 이름이 아니라 플레이어가 이해할 원인으로 작성한다.

```text
[1라운드 1묶음 · 2수]
강호낭인이 속공을 사용했다.
상대는 오른쪽 거리 1에 있었다.
속공이 이동 전에 적중해 피해 6을 주었다.
```

표시 대상:

- 행동 배치·해제·배치 불가
- 대상 지정·지정 실패
- 자원 부족
- 대응 적용
- 공격 적중·빗나감
- 이동 성공·충돌
- 체력·기력·내력·기세 변화
- STEP 11 이후 행동 중단과 원인

과거의 별도 합 수치 비교는 현재 POC에서 사용하지 않으므로 `합 예상` UI를 표시하지 않는다.

## 11. 대응 표현

### 막기

- 같은 수 50% 감소와 같은 묶음 방어도 감소를 비교한다.
- 실제 적용된 높은 감소 효과와 최종 피해를 로그에 표시한다.

### 회피

- 같은 수 완전 회피 성공 여부를 표시한다.

### 태세 결합

- `태세+막기`: 묶음 전체 방어와 방어도 50% 증가
- `태세+회피`: 묶음 전체 완전 회피
- 결합 비용을 합산해 표시한다.

RESPONSE 10.6은 구현됐으나 최신 Windows 실제 판정 확인 전까지 `IMPLEMENTED_FOR_REVIEW`다.

## 12. STEP 11 예정 표현

피격 중단·집중·강건은 아직 현행 UI 완료 항목이 아니다.

구현 시 필요한 상태:

- 준비 중
- 실행 완료
- 피해로 중단
- `[집중]`으로 중단 방지
- `[강건]`으로 중단 방지
- 사망·강제 제어로 취소

현재 UI에 중단이 이미 작동하는 것처럼 표시하지 않는다.

## 13. 접근성

- 색과 함께 텍스트·아이콘·형태를 사용한다.
- 키보드 포커스 순서는 카드 트레이→현재 행동 슬롯→대상 칸→진행→상세→기록을 따른다.
- 포인터 없이도 카드 선택·슬롯 배치·대상 선택·해제가 가능해야 한다.
- 점멸·화면 흔들림·빠른 이동 연출을 줄이는 옵션을 고려한다.
- 긴 한국어 문구와 최소 해상도에서 정보가 잘리지 않아야 한다.
- 자산 누락 시 기능명·상태 문구·기본 도형으로 대체한다.
- 접근성 검수는 실제 사용자 입력 확인 전까지 `NOT_RUN`이다.
