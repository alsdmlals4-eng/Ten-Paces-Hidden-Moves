# 십보강호 전투 UI·아트 명세

> 책임: 현재 T0 전투 화면·카드·행동 계획·상태·접근성 표현 계약  
> 판정 원본: `docs/02_COMBAT_RULES.md`  
> POC 범위: `docs/05_COMBAT_POC_SPEC.md`  
> 연출 흐름: `docs/10_COMBAT_PRESENTATION_PLAN.md`

## 1. 상태 구분

- `CURRENT`: 실제 Godot UI에 존재하고 정적 계약이 연결된 상태.
- `USER_CONFIRMED`: 사용자가 Windows에서 확인한 상태.
- `IMPLEMENTED_FOR_REVIEW`: 코드에는 있으나 최신 사용자 확인이 남은 상태.
- `PLANNED`: 후속 STEP에서 구현할 상태.
- `ACCESSIBILITY_TARGET`: 접근성 목표이며 아직 실제 입력 검증 전 상태.

목표 문구와 현재 화면 문구를 섞지 않는다.

## 2. UI 목표

플레이어가 한 화면에서 다음 질문에 답할 수 있어야 한다.

1. 양측은 어디에 있고 현재 거리는 얼마인가?
2. 체력·기력·내력·절초 기세는 얼마인가?
3. 지금 어느 라운드·행동 묶음·수인가?
4. 어떤 행동을 어느 수에 배치했는가?
5. 이동 목적지와 공격 방향은 지정됐는가?
6. 진행할 수 없다면 무엇이 부족한가?
7. 판정 결과가 왜 발생했는가?

전투 전 상대 정보는 무협식 표현을 사용할 수 있지만, 전투 중 실패 원인과 판정 결과는 직접적이고 반복 가능한 용어를 사용한다.

## 3. 화면 구조

| 영역 | 현재 표시 |
|---|---|
| 상단 좌측 | 플레이어 기본 초상 도형·이름·이명·체력·기력·내력·상태 |
| 상단 중앙 좌측 | 플레이어 절초 기세, 최대 5칸 |
| 상단 중앙 | 라운드·행동 묶음·현재 선택 수·판정 순서 |
| 상단 중앙 우측 | 상대 절초 기세, 최대 5칸 |
| 상단 우측 | 상대 기본 초상 도형·이름·이명·체력·기력·내력·상태 |
| 중앙 | 1~10번 전장, 플레이어 4번·상대 7번 시작 |
| 하단 상단 | `3수 → 3수 → 4수` 행동 슬롯과 진행 버튼 |
| 하단 하단 | 기초 행동 8종 |
| 좌측 오버레이 | 카드 상세 패널 |
| 우측 오버레이 | 접이식 전투 기록 |

- 전장과 캐릭터가 화면의 중심이다.
- 시작 거리는 3칸이다.
- 좌우 세로 상태 패널을 두어 전장 너비를 줄이지 않는다.
- 하단 카드는 무작위 손패가 아니라 현재 사용 가능한 행동 목록이다.
- 덱·드로·버린 카드·남은 카드 수를 표시하지 않는다.

## 4. 전장 칸 상태

색뿐 아니라 형태·테두리·문구를 함께 사용한다.

| 상태 | 의미 | 현재 입력 |
|---|---|---|
| 기본 | 선택과 무관한 칸 | 포인터 입력을 가로채지 않음 |
| 플레이어 | 플레이어 점유 | 대상 선택 불가 |
| 상대 | 상대 점유 | 이동 대상으로 선택 불가 |
| 이동 가능 | 현재 이동의 유효 목적지 | 클릭 가능 |
| 공격 가능 | 현재 공격의 좌·우 방향 후보 | 클릭 가능 |
| 선택 | 확정한 목적지·방향 | 슬롯 문구로도 확인 |
| 비활성 | 전장 밖·점유·범위 밖 | 클릭 불가 |

- 이동 후보는 녹색 계열과 이동 표식을 사용한다.
- 공격 후보는 적색 계열과 공격 표식을 사용한다.
- 공격 후보 칸은 공격의 실제 피해 대상 칸이 아니라 방향 선택 인터페이스다.
- 공격은 상대를 자동 추적하지 않는다.

## 5. 상단 HUD

고정 순서:

```text
플레이어 상태 | 플레이어 절초 기세 | 라운드 | 상대 절초 기세 | 상대 상태
```

### 5.1 상태 패널

- 체력·기력·내력을 `현재/최대`로 표시한다.
- 기본 초상은 현재 벡터 도형 placeholder다.
- 상태가 있으면 작은 상태 칩으로 표시한다.
- 기력·내력 계획 예상값은 배치가 바뀔 때 HUD에 반영한다.
- 예상값은 실제 판정 전 상태라는 사실을 색·문구 또는 별도 강조로 구분해야 한다.

### 5.2 절초 기세

- 정확히 5칸이다.
- 현재 fixture의 플레이어 4/5·상대 3/5는 게이지 표현 검증값이다.
- 정식 전투 시작 기세·절초 소모 UI는 아직 완료되지 않았다.

### 5.3 라운드·묶음

상단 중앙 실제 형식:

```text
제 1 라운드
행동 묶음  1/3
1수 · 2수 · 3수 선택
대응 → 속공 → 이동 → 일반 공격
```

하단 행동 슬롯 영역의 진행 표기는 `라운드 1`만 사용한다. `(5/10수)` 같은 중복 진행 문구는 사용하지 않는다.

## 6. 행동 슬롯

### 6.1 구조

```text
1묶음: 1수 · 2수 · 3수
2묶음: 4수 · 5수 · 6수
3묶음: 7수 · 8수 · 9수 · 10수
```

### 6.2 상태

- 지나간 수.
- 현재 편집 가능.
- 행동 점유.
- 대상 지정 대기.
- 자원 부족.
- 미래 묶음 잠금.
- 판정 완료.

### 6.3 배치

- 1슬롯 행동은 한 수를 점유한다.
- 2슬롯 행동은 현재 묶음의 연속된 두 수를 점유한다.
- 묶음 경계를 넘는 배치는 거부한다.
- 다중 슬롯의 어느 점유 칸을 눌러도 행동 전체를 해제한다.
- 실제 실행 수는 마지막 점유 수다.

### 6.4 슬롯 문구

- 카드 이름.
- 다중 슬롯 연결.
- 시작 위치 기준 이동 예: `← 3번`, `→ 5번`.
- 공격 방향: `← 공격`, `→ 공격`.
- 대상 미지정.
- 자원 부족.
- 결합 행동: `태세+막기`, `태세+회피`.

## 7. 행동 선택·대상 지정

현재 포인터 흐름:

```text
카드 클릭
→ 빈 행동 슬롯 클릭
→ 필요한 경우 전장 칸 클릭
→ 슬롯 문구·예상 자원 확인
```

- 카드 클릭은 상세 고정과 행동 선택을 함께 수행한다.
- 대상 지정 중에는 입력 순서가 꼬이지 않도록 다른 카드 선택을 제한한다.
- 지정이 끝나면 다음 미지정 행동으로 자동 이동할 수 있다.
- 빈 전장 클릭은 선택·고정 상세를 해제한다.

### 이동

- 시작 위치 4번에서 기본 이동은 3번 또는 5번을 후보로 표시한다.
- 보법은 좌·우 1칸과 2칸을 표시하되 상대 점유 칸과 전장 밖을 제외한다.
- 앞선 이동을 반영한 예상 위치를 이후 대상 지정 기준으로 사용한다.

### 공격

- 속공은 좌·우 거리 1까지 방향 후보를 표시한다.
- 강공은 좌·우 거리 1·2까지 방향 후보를 표시한다.
- 슬롯에는 실제 선택 방향만 기록한다.
- 시작 거리 3에서는 이동 없이 속공·강공이 적중하지 않는다는 사실을 방향·거리 로그로 설명한다.

## 8. 진행 버튼

실제 구성:

```text
상단 캡션: 행동 묶음 판정
버튼: 진행
하단 상태 문구
```

실제 상태 문구:

| 조건 | 상태 문구 |
|---|---|
| 배치 미완성·대상 미지정·자원 부족 | `행동·대상·자원 확인 필요` |
| 현재 묶음 완료 | `배치 완료` |
| 동기 판정과 다음 묶음 반영 후 | `판정 완료` |

- 진행 버튼은 현재 묶음의 슬롯·대상·예상 자원이 모두 유효할 때만 활성화한다.
- 현재 판정은 신호 호출 안에서 동기적으로 완료된다.
- 별도의 눈에 보이는 `판정 중` 애니메이션 상태나 단계별 입력 잠금 화면은 아직 없다.
- 진행 신호가 처리되는 동안 같은 이벤트 루프에서 중복 클릭할 수 없으며, 판정 후 다음 묶음 상태로 즉시 전환한다.
- 후속 단계에서 판정을 시간차 연출로 재생할 경우 명시적 `resolving` 상태와 입력 잠금이 필요하다.

## 9. 카드 템플릿

### 9.1 고정 정보 계층

| 위치 | 정보 |
|---|---|
| 좌측 상단 | 소속 `[기초]` 또는 문파·세력 |
| 상단 중앙 | 사거리 |
| 우측 상단 | 기능 배지: 이동·공격·대응·회복·강화 |
| 제목부 | 카드 이름 |
| 중앙 | 카드 원화 |
| 하단 | 행동 슬롯·기력·내력 |

- `[속공]`, `[집중]`, `[강건]`, `[검법]`, `[보법]`은 상세 태그다.
- 행동력은 사용하지 않는다.
- 공통 `막기 경감` 필드를 카드 하단에 두지 않는다.
- 프레임·소속·사거리·기능 배지·비용·원화·문구·데이터를 분리한다.
- 생성 이미지에 UI 텍스트를 구워 넣지 않는다.

### 9.2 기초 행동 8종

| 행동 | 기능 | 대상 표시 |
|---|---|---|
| 이동 | 이동 | 좌·우 1칸 녹색 목적지 |
| 보법 | 이동 | 좌·우 1~2칸 녹색 목적지 |
| 막기 | 대응 | 자신, 대상 선택 없음 |
| 회피 | 대응 | 자신, 대상 선택 없음 |
| 속공 | 공격 | 좌·우 거리 1 붉은 방향 |
| 강공 | 공격 | 좌·우 거리 1~2 붉은 방향 |
| 명상 | 회복 | 자신, 대상 선택 없음 |
| 태세 | 강화 | 자신 또는 대응 결합 |

## 10. 카드 상세

### 임시 보기

- 마우스를 올리면 표시한다.
- 마우스를 벗어나면 닫는다.

### 고정 보기

- 카드를 클릭하면 고정한다.
- 같은 카드 재클릭 또는 빈 전장 클릭으로 닫는다.
- 행동 선택 상태와 상세 고정 상태를 별도 테두리·메타 상태로 구분한다.

표시 순서:

1. 이름·소속·기능·사거리.
2. 대상.
3. 피해·이동량·방어도·회복 등 핵심 수치.
4. 사용 조건.
5. 효과 설명.
6. 태그.
7. 행동 슬롯·기력·내력.
8. 플레이버 문구.

## 11. 전투 기록

- 기본적으로 접힌 `기록` 버튼이다.
- 펼치면 오른쪽 오버레이에 로그를 표시한다.
- 현재 구현은 로그 문자열 추가·초기화·수동 스크롤을 지원한다.
- 새 로그 추가 후 자동으로 최하단을 유지하거나, 사용자가 과거 로그를 읽을 때 자동 이동을 멈추는 정책은 아직 구현하지 않았다.
- 개발 이벤트 이름보다 플레이어가 이해할 원인을 우선한다.

표시 대상:

- 행동 배치·해제·배치 불가.
- 대상 지정.
- 자원 부족.
- 대응 준비·적용.
- 공격 적중·방향 실패·사거리 실패.
- 이동 성공·실패·충돌.
- 체력·기력·내력·기세 변화.
- STEP 11 이후 행동 중단과 원인.

예시:

```text
[1라운드 1묶음 판정]
[1수] 강호낭인이 4번에서 5번으로 이동했다.
상대는 7번에서 6번으로 이동했다.
[3수] 강호낭인이 오른쪽으로 속공했다.
거리 1에서 적중했다.
```

현재 POC에는 별도 합 수치 비교가 없으므로 `합 예상` UI를 표시하지 않는다.

## 12. 대응 표현

### 막기

- 같은 수 50% 감소 후보와 같은 묶음 방어도 후보를 비교한다.
- 선택된 감소 효과·감소량·최종 피해를 로그에 표시한다.

### 회피

- 같은 수 완전 회피를 직접 문구로 표시한다.

### 태세 결합

- `태세+막기`: 묶음 전체 방어와 방어도 4→6.
- `태세+회피`: 묶음 전체 완전 회피.
- 결합 비용을 예상 자원과 카드 상세에 합산한다.

RESPONSE 10.6 UI 연결은 구현됐으나 최신 Windows 실제 판정 확인 전까지 `IMPLEMENTED_FOR_REVIEW`다.

## 13. STEP 11 예정 표현

피격 중단·집중·강건은 아직 현행 UI 완료 항목이 아니다.

필요 상태:

- 준비 중.
- 실행 완료.
- 피해로 중단.
- `[집중]`으로 중단 방지.
- `[강건]`으로 중단 방지.
- 사망·강제 제어로 취소.

현재 UI에 중단이 이미 작동하는 것처럼 표시하지 않는다.

## 14. 입력·접근성

### 현재 입력

- 카드·슬롯·대상·진행·기록은 포인터 조작 기준이다.
- 현재 주요 Button은 키보드 포커스를 받지 않도록 구성돼 있다.
- 따라서 키보드 전체 흐름은 구현 완료가 아니다.

### ACCESSIBILITY_TARGET

- 카드 트레이→현재 행동 슬롯→대상 칸→진행→상세→기록 순의 명확한 포커스 이동.
- 포인터 없이 카드 선택·슬롯 배치·대상 선택·해제 가능.
- 색 외 텍스트·아이콘·형태 제공.
- 긴 한국어 문구와 최소 해상도에서 잘림 방지.
- 점멸·화면 흔들림·빠른 모션 감소 옵션.
- 자산 누락 시 기능명·상태 문구·기본 도형 폴백.

실제 접근성 사용자 검수는 `NOT_RUN`이다.

## 15. 현재 구현·검증 상태
