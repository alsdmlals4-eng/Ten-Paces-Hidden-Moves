# 십보강호 활성 컨텍스트

> 목적: 작업자가 바뀌어도 과거 대화 없이 현재 방향, 확정 규칙, 미확정 범위와 다음 작업을 복원하게 한다. 모든 작업 종료 시 이 문서를 갱신한다.

## 1. 현재 상태

- 현재 단계는 **전투 POC 기반 UI와 10칸 전투판 구현**이다.
- 구현 브랜치는 `agent/t0-combat-poc-board`, 스택형 Draft PR #7이다.
- STEP 0 카드 UI 컴포넌트·JSON·분리 SVG 자산은 구현됐고 런타임 검증은 남아 있다.
- STEP 1 실제 10칸 Godot 전투판 씬을 구현했다.
- STEP 2 플레이어 3번 / 상대 8번 / 동일 스케일 / 높이 1.5칸 폭 / 발 앵커를 구현했다.
- Godot 기본 실행 장면은 `scenes/combat/combat_board_preview.tscn`이다.
- Windows 직접 실행 진입점은 `tools/run_combat_board.cmd`다.
- 다음 구현은 STEP 3 전투 배경이다.
- `docs/[백업]/`은 과거 기록, `docs/[보류]/`는 재개 전 구현 금지 영역이다.

## 2. 제품 방향

- 장르: 1대1 무협 로그라이트 비무 대회.
- 핵심 경험: 상대의 거리·자원·성향을 읽고 `3수 → 3수 → 4수` 행동 묶음을 계획하는 예측전.
- 내부 시스템은 수치와 데이터로 정확히 계산한다.
- 플레이어에게는 불필요한 개발용 분류를 직접 노출하지 않고 무림식 이명·풍문·전적·정탐 문장으로 암시한다.
- 같은 표현은 같은 의미를 가져 반복 플레이로 학습 가능해야 하며 강한 적을 약한 표현으로 속이지 않는다.
- 연출은 규칙을 숨기는 장식이 아니라 상대 읽기, 위험 선택, 예측, 결과 원인, 성장 보상을 이해시키는 프레젠테이션 계층이다.
- 프레젠테이션은 피해·보상·수련·저장을 결정하지 않는다.

## 3. 고정 숫자 구조

- 전장: 10칸.
- 라운드: 행동 타이밍 10개.
- 행동 묶음: `3수 → 3수 → 4수`.
- 전체 대회: 10전, 8전 8강·9전 준결승·10전 결승.
- 데모: 1~5전.
  - 1~2전: 내부전.
  - 3~4전: 예선.
  - 5전: 예선 결승이자 데모 최종전.

## 4. 전투 핵심

- 같은 타이밍 기본 해상 순서는 `대응 → 속공 → 이동 → 일반 공격`이다.
- `[속공]`은 이동보다 먼저 해상한다.
- 같은 우선순위의 유효 공격은 `[합]`이 발생할 수 있다.
- 실제 직접 피해를 받으면 아직 실행되지 않은 이동·공격·다중 슬롯 행동은 기본적으로 중단된다.
- `[집중]`과 `[강건]`은 강제 제어·사망을 제외한 중단 방지 계약이다.
- 공용 전투 해상과 구조화 이벤트가 결과를 소유하며 UI·VFX·애니메이션은 이를 표현한다.

## 5. 무공·심법

- 각 핵심무공 종류는 관련 문파 전용 1종과 공용 2종을 기본 콘텐츠 단위로 한다.
- 핵심무공은 여러 개 보유 가능하다.
- 심법은 보유 자체가 최대 2개다.
- 시작 문파 핵심무공과 심법은 각각 3성이다.
- 성급 `n`에 도달하는 비용은 `n`포인트다.
- 핵심무공은 3·6·9성에 기술, 10성에 절초를 얻는다.
- 심법은 3·6·9성에 패시브·기믹 생성/강화·명상 효과 등을 조합하고 10성에 진의를 얻는다.
- 핵심무공 10성은 최대 1개, 심법 10성은 최대 1개다.
- 심법은 관련 핵심무공을 단방향으로 강화한다.
- 화산의 `[연격]` 같은 핵심 기믹은 심법 3성에서 생성하고 10성 진의에서 운용을 강화한다.
- 문파 효과는 가능한 한 공용 비용·합·사거리·상태·명상·연계 수정자로 표현한다.

### 5.1 절초 해금과 전투 발동 UX

상태: **구조 승인 / 정확한 수치·명칭 미확정 / 이후 기획 작업**.

절초는 두 개의 서로 다른 조건을 가진다.

```text
핵심무공 10성 → 절초 자체 해금
전투 중 기세 충전 → 해당 전투에서 절초 발동 가능
```

승인된 방향:

- 플레이어와 상대의 절초 기세를 전투 화면 상단 HUD에 각각 표시한다.
- 짧은 칸형 게이지, 절초 아이콘과 발동 가능 상태를 사용한다.
- 라운드 진행, 합, 회피, 막기 등 전투 참여와 판단 결과가 기세 축적 후보가 된다.
- 점수 획득 시 증가량뿐 아니라 `합`, `회피` 같은 이유를 함께 표시한다.
- 조건 미달 시 절초 행동의 잠금 이유를 표시한다.
- 조건 달성 시 상단 HUD와 절초 행동에서 `절초 가능` 상태를 명확히 강조한다.
- 절초를 행동 슬롯에 배치하면 발동 예약 상태를 표시하고 정상 사용 뒤 기세를 소모한다.
- UI·VFX·오디오는 절초 해금·누적·소모를 직접 계산하지 않는다.

## 6. 성과·행운·수련

### 고정 정산식

```text
최종 수련 포인트 = 승리 기본 5 + 등급 보너스 + 행운 앞면 수
```

### 등급 보너스

| 등급 | 보너스 |
|---|---:|
| C | +1 |
| B | +3 |
| A | +5 |
| S | +7 |

- A는 의도한 평균적 전투 수행 등급이다.
- 문파별 기믹 수행도 평가는 작업량 때문에 현재 `[보류]`다.

### 행운 결과표

행운 다이스 3개는 비공개로 굴린다. 독립 확률 3회가 아니라 등급별 `앞면 0~3개` 결과표에서 한 번 추첨한다.

| 등급 | 0개 | 1개 | 2개 | 3개 | 평균 행운 |
|---|---:|---:|---:|---:|---:|
| C | 0% | 0% | 5% | 95% | 2.95 |
| B | 5% | 5% | 5% | 85% | 2.70 |
| A | 20% | 5% | 20% | 55% | 2.10 |
| S | 60% | 10% | 15% | 15% | 0.85 |

플레이어 표시는 `B`, `B+`, `B++`, `B+++`처럼 앞면 수만 `+`로 붙인다. 개별 다이스 면은 공개하지 않는다.

## 7. 전투 UI 고정 방향

- 전투판은 정확히 10칸이다.
- 캐릭터는 정확히 한 칸을 점유하며 발 중심이 칸 중앙 앵커와 일치한다.
- 시작 배치는 플레이어 3번, 상대 8번이다.
- 카메라는 10칸과 양측 전신을 함께 보여주는 고정 와이드다.
- 화면 상단은 플레이어 상태·플레이어 절초 기세·라운드 묶음·상대 절초 기세·상대 상태 순서다.
- 화면 하단 위쪽은 현재 행동 묶음 슬롯과 `[진행]`이다.
- 화면 하단 아래쪽은 보유·사용 가능 기술 전체다.
- 덱·손패·드로우·버림 더미는 사용하지 않는다.

## 8. STEP 0 카드 UI 계약

- 카드 상단은 `소속 배지 / 사거리 / 기능 배지`다.
- 기능 배지는 이동·공격·대응·회복·강화 5종이다.
- 기초 행동의 소속은 `[기초]`다.
- 카드 하단은 행동 슬롯·기력·내력만 사용한다.
- 행동력과 공통 `막기 경감`은 사용하지 않는다.
- 기초 행동은 이동·막기·회피·속공·강공·명상·태세 7종이다.
- `CardView`, `CardDetailPanel`, `CardCatalog`를 분리한다.
- 원화·배지·비용 아이콘은 독립 자산이며 텍스트와 수치는 엔진 데이터로 표시한다.

## 9. STEP 1·2 구현 결과

- `scenes/combat/combat_board_preview.tscn` 추가.
- `Tile01..Tile10` 실제 타일 10개 생성.
- 플레이어 3번, 상대 8번 배치.
- 양측 동일 스케일, 높이 타일 폭의 1.5배.
- 발 중심과 타일 공통 앵커 계산 일치.
- 타일과 캐릭터를 독립 재사용 컴포넌트로 분리.
- 프로젝트 실행(F5) 시 전투판으로 바로 진입.
- `tools/run_combat_board.cmd` 더블클릭 시 전투판 게임 창 직접 실행.
- `tools/verify_and_commit_step0.cmd`는 STEP 0·1·2 통합 검증·PASS 보고서·commit·push 용도다.

## 10. 즉시 다음 작업

1. 사용자 Windows에서 `tools/run_combat_board.cmd` 또는 Godot F5로 전투판을 직접 확인한다.

3. STEP 3 전투 배경을 제작한다.
4. STEP 4 상단 HUD를 구현한다.
5. STEP 5 `3수·3수·4수` 행동 슬롯을 구현한다.

## 11. 보호 범위

- 승인 카드 템플릿과 원화 방향.
- 10칸 전장과 `3수·3수·4수` 구조.
- 기초 카드 7종과 기능 배지 5종.
- 행동 슬롯·기력·내력 비용 체계.
- `docs/[백업]/`, `docs/[보류]/`, 기존 Plan·PR 이력.
- 실행되지 않은 검증의 미검증 상태.

## 12. 주요 위험

- Windows 작업본에 원격보다 최신 파일 또는 미커밋 변경이 있을 수 있다.
- 현재 대화 환경에서는 사용자 PC의 Godot 화면을 직접 조작하거나 렌더 결과를 직접 확인하지 못한다.
- Actions 성공은 Godot·Windows 수동 플레이테스트 또는 Required Check 강제를 의미하지 않는다.

## 13. 완료 판정

STEP 0과 STEP 1·2는 저장소 구현·정적 계약·자동 검증 코드까지 완료했다. 사용자 Windows에서 실제 씬 로드·렌더·입력·앵커 검증은 `NOT_RUN`이다.
