# 십보강호 협업 규칙

## 1. 규칙 우선순위

문서와 실제 파일이 충돌하면 다음 순서를 따른다.

1. 사용자의 최신 확정 지시
2. 이 `AGENTS.md`
3. 다섯 활성 기획서
4. `docs/ACTIVE_CONTEXT.md`
5. `docs/DOCUMENTATION_MAP.md`
6. 연결된 상세 명세·부록
7. 현재 Issue·Goal·Plan
8. 실제 구현 파일과 실행 결과

실제 구현 상태는 파일과 실행 결과로 확인한다. 문서만 보고 구현 완료를 추정하지 않는다.

## 2. 작업 시작 컨텍스트

사용자와 작업자가 처음 보는 활성 기획서는 정확히 다섯 개다.

1. `[기획서]/01_게임기획/게임기획서.md`
2. `[기획서]/06_프로그래밍/프로그래밍_기획서_로드맵_MVP.md`
3. `[기획서]/04_아트디자인/아트_기획서.md`
4. `[기획서]/07_사운드/사운드_기획서.md`
5. `[기획서]/08_QA_PM/QA_PM_기획서.md`

작업자는 다음 순서로 현재 상태를 확인한다.

```text
최신 사용자 지시
→ AGENTS.md
→ 다섯 활성 기획서
→ docs/ACTIVE_CONTEXT.md
→ docs/DOCUMENTATION_MAP.md
→ 현재 작업과 연결된 상세 명세·부록
→ 실제 수정 대상 파일·자산·테스트
```

- 기존 `docs/`와 세부 `[기획서]/` 문서는 활성 기획서가 아니라 상세 명세·부록이다.
- 저장소 전체를 무조건 읽지 않고 활성 기획서의 부록 링크와 문서 지도를 사용한다.
- `docs/[백업]/`과 `docs/[보류]/`는 사용자가 재개하지 않는 한 기본 읽기·구현 대상에서 제외한다.
- Active Context가 실제 파일·활성 기획서와 충돌하면 차이를 보고하고 같은 작업에서 정리한다.

## 3. 활성 기획서 품질 기준

### 사용자 가독성

- 첫 화면에서 프로젝트 한 줄 설명, 현재 상태, 핵심 결정과 다음 작업이 보이게 한다.
- 제목, 표, 짧은 문단과 상태 구분을 사용한다.
- 내부 용어를 처음 사용할 때 설명한다.
- 다른 문서를 열지 않아도 핵심 방향을 이해할 수 있게 한다.

### 새 작업자의 재현 가능성

- 과거 대화를 볼 수 없다고 가정한다.
- 작품의 핵심 경험, 품질 기준, 확정 규칙, 현재 상태, 파일 경로, 금지 사항, 완료 기준과 다음 작업을 상세히 기록한다.
- 문서만 읽고 기존과 같은 방향과 품질을 만들 수 없다면 문서가 부족한 것으로 본다.

### 최신 이미지 유지

- 최신 이미지의 책임 원본은 `[기획서]/04_아트디자인/아트_기획서.md`와 `이미지_인덱스.md`다.
- 다른 문서는 이미지를 복제하지 않고 이 원본을 링크한다.
- 이미지 생성·수정·승인·폐기 시 두 문서를 같은 작업에서 갱신한다.
- 승인 원본이 저장소에 없으면 `원본 반입 필요`로 기록한다.
- 이미지 속 한글·수치·분류는 기준 데이터가 아니다. 편집 가능한 UI 텍스트와 아이콘 자산이 소유한다.

## 4. 팀별 문서 책임

- 규칙·시나리오·전투·성장·UI/UX·밸런스 → 게임 기획서.
- 엔진·코드·AI·데이터·로드맵·MVP·기술 위험 → 프로그래밍 기획서.
- 캐릭터·배경·카드·문양·아이콘·애니메이션·VFX → 아트 기획서와 이미지 인덱스.
- BGM·SFX·보이스·오디오 이벤트·믹싱 → 사운드 기획서.
- 테스트·일정·범위·위험·승인·인수인계 → QA·PM 기획서.

복수 분야 작업은 관련된 활성 기획서를 모두 갱신한다.

## 5. 기존 상세 문서와 중복 관리

- 기존 `APPROVED_*`, `docs/`, 세부 `[기획서]/`, `plans/`는 다섯 활성 기획서를 증명하는 상세 명세·부록으로 취급한다.
- Documentation Map 첫 화면에는 다섯 활성 기획서만 보여준다.
- 각 활성 기획서 안에서 필요한 상세 명세를 부록으로 연결한다.
- 중복 내용은 다섯 활성 기획서로 흡수한다.
- 고유 수치·데이터·계약·테스트가 없는 문서만 참조 검사를 거쳐 아카이브하거나 제거한다.
- 활성 문서를 `v2`, `final`, `latest` 또는 날짜별 복제본으로 만들지 않는다.
- 이전 내용은 Git 이력으로 보존하고 현재 기준 파일을 직접 갱신한다.

## 6. 기획 변경과 동기화

- 대화에서 확정된 변경은 같은 작업 흐름에서 관련 활성 기획서와 상세 명세에 반영한다.
- 수치·용어가 여러 문서에 걸치면 단일 책임 원본을 먼저 정하고 나머지는 참조한다.
- 확정, 진행 중, 검증 필요, 보류를 구분한다.
- 범위·완료 기준·데이터 구조가 바뀌면 테스트 체크리스트와 Plan을 함께 확인한다.
- 프로젝트 첫 화면의 설명·링크·현재 범위가 바뀌면 README와 manifest를 갱신한다.
- 폐기된 표현·수치·구조는 활성 문서에서 제거한다.
- 보류 항목은 사용자가 재개하기 전 코드·씬·데이터·자산·테스트에 반영하지 않는다.

## 7. 구현·표현 책임 분리

- 전투 도메인이 피해, 위치, 자원, 합, 성장, 저장 결과를 소유한다.
- UI·애니메이션·VFX·사운드는 구조화 이벤트를 표현한다.
- 아트 시안의 임시 숫자와 문구가 전투 데이터를 변경하지 않는다.
- 무공 이름·분류·성급·사거리·비용은 데이터와 편집 가능한 UI 레이어가 소유한다.
- 생성 이미지의 잘못된 한글을 기준으로 문서를 수정하지 않는다.

## 8. 누락 방지 자동 검사 — 필수

다음 검사를 통과하지 못하면 작업을 완료로 보고하지 않는다.

```bash
python scripts/check_planning_sync.py
python tests/static_check.py
godot --headless --path . --script res://tests/run_all.gd
```

- Godot 실행 환경이 없으면 앞의 두 검사를 수행하고 런타임 미검증을 기록한다.
- `check_planning_sync.py`는 정확히 다섯 활성 기획서, README·Documentation Map 노출, manifest, 이미지 인덱스, 링크와 구현 경로를 검사한다.
- PR에는 `.github/pull_request_template.md`의 분야별 기획서와 검증 항목을 확인한다.

## 9. 작업 종료 컨텍스트 정비 — 필수

1. 이번 작업의 확정 결정과 변경 수치를 관련 활성 기획서에 반영한다.
2. 연결된 상세 명세·부록과 실제 구현·자산을 갱신한다.
3. 이미지 작업이면 아트 기획서와 이미지 인덱스를 갱신한다.
4. `[기획서]/manifest.json`, `[기획서]/README.md`, `docs/ACTIVE_CONTEXT.md`, `docs/DOCUMENTATION_MAP.md`의 영향을 확인한다.
5. 폐기된 표현·수치·구조를 검색하고 충돌을 제거한다.
6. 보류한 내용은 활성 구현에서 분리한다.
7. 실제 구현·테스트 상태와 문서가 일치하는지 확인한다.
8. 자동 검사와 가능한 런타임·수동 검증을 수행한다.
9. 수행하지 못한 검증과 남은 위험을 기록한다.
10. GitHub 반영 결과와 다음 작업을 보고한다.

단순 대화라도 프로젝트 방향·수치·용어·범위·이미지 방향이 확정되었다면 문서 갱신 대상이다.

## 10. Base 자동 승격

- 안정적이고 여러 프로젝트에 적용 가능한 작업 규칙은 공용 Base 승격을 검토한다.
- 프로젝트 고유 이름, 밸런스 수치, 세계관, 카드 사양, Godot 경로와 일회성 세부는 프로젝트 문서에만 남긴다.
- Base에도 새 버전 파일을 만들지 않고 기존 기준 파일을 직접 갱신한다.

## 11. 작업 결과 보고

GitHub 갱신 후 다음을 보고한다.

- 갱신한 활성 기획서와 변경 이유.
- 연결된 상세 명세·이미지 인덱스·manifest 갱신 결과.
- 제거하거나 부록으로 강등한 중복 내용.
- 실제 수행한 검증과 수행하지 못한 검증.
- 남은 위험과 다음 작업.
