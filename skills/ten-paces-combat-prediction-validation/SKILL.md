# Ten Paces Combat Prediction Validation

예측전의 신뢰는 계산 정확도만으로 생기지 않는다. 플레이어가 **상대가 먼저 같은 정보로 두 수를 잠갔음**, **내 선택이 바뀌지 않았음**, **1수 결과가 2수에 어떻게 이어졌는지**, **합·회피·막기·절초 기세가 왜 변했는지**를 재현 가능하게 이해해야 한다.

## Contract

다음 사실을 하나의 표로 고정한다.

| 단계 | 정본 소유자 | 플레이어에게 보여 줄 증거 | 금지 |
|---|---|---|---|
| 상대 2수 결정 | 전투 도메인 / AI 입력 스냅샷 | 사전 잠금 상태와 공개 가능한 근거 | 플레이어 선택 뒤 상대 행동 재굴림 |
| 플레이어 2수 잠금 | 전투 도메인 | 슬롯·잠금·변경 불가 상태 | UI만 잠기고 데이터가 변경 가능한 상태 |
| 동시 공개 | 구조화 전투 이벤트 | 양측 1수·2수와 예약 상태 | 순서를 숨겨 결과를 사후 조작한 인상 |
| 1수·2수 해상 | 공용 해상기 | 유효/무효 이유, 위치·상태 변화 | 1수 뒤 2수 선택 재작성 |
| 합과 피해 | 공용 계산 | 합 성립·최종 합·차이·피해 이유 | VFX 또는 HUD가 피해·보상을 직접 결정 |
| 절초 기세 | 전투 상태 | 획득·상한·사용 가능·예약·소모 이유 | HUD가 기세를 계산하거나 미해금 행동을 숨김 |

정확한 기세 수치가 미확정이면 `PROPOSED`로 유지하고 구현하지 않는다. 명칭, 상한, 획득량, 유지·초기화, AI 사용 기준, 취소·실패 소모는 승인 전 빈칸을 수치로 채우지 않는다.

## Design and implementation sequence

1. 현재 규칙·UI·도메인·테스트에서 같은 사실이 서로 다르게 쓰인 지점을 찾는다.
2. 상태 전이를 `ready → opponent_locked → player_locked → reveal → resolve_first → resolve_second → round_end`로 명시하고, 각 전이의 입력·출력·불변식을 적는다.
3. 전투 이벤트를 계산 결과의 유일한 설명 원본으로 두고 HUD·로그·연출은 이를 소비하게 한다.
4. 행동 무효, 사거리 이탈, 상태이상, 동시 공격, 동률 합, 절초 미해금·미충전·예약·취소를 정상 흐름과 같은 수준으로 설계한다.
5. 전투 종료의 보상·수련은 전투 판정과 분리하되, 합·제약·성과의 원인을 결과 화면에서 추적 가능하게 한다.
6. 문서·Plan·테스트·Active Context의 이전 수치·구형 2수 규칙·폐기된 행동력 표현을 검색해 현행 원본으로 정리한다.

## Required verification

- 같은 시드·공개 정보에서 AI 잠금 결과가 재현된다.
- 플레이어 잠금 뒤 입력·저장/불러오기·UI 갱신이 행동을 바꾸지 않는다.
- 1수 결과가 2수의 유효성에 영향을 주어도 2수 선택 자체는 바뀌지 않는다.
- 합·회피·막기·피격·절초 기세의 표시는 구조화 이벤트와 일치한다.
- 미해금·미충전 절초는 이유와 다음 조건을 보여 주며 일반 경로를 막지 않는다.
- 1280×720과 기본 입력에서 양측 잠금·예약·결과 이유가 읽힌다.
- 규칙·UI·저장·보상·테스트의 활성 참조에 구형 수치나 용어가 남지 않는다.

## Output

`규칙 상태 / 전이표 / 표시 증거 / 변경 파일 / 정상·실패·경계 검증 / 미확정 수치 / 남은 사람 플레이 검증`을 분리해 보고한다.

## Failure conditions

- AI 잠금·플레이어 잠금·공개·해상의 시간 순서를 추측으로 채운다.
- HUD 또는 연출이 전투 상태를 소유한다.
- 절초 기세를 숫자만 보여 주고 획득·잠금·소모 이유를 설명하지 않는다.
- 자동 테스트 통과를 실제 예측전의 이해 가능성 검증으로 표시한다.
